## 1) Objective of Day 23

After stabilizing the RBAC layer and the dossier review workflow, the application still lacked a practical correction path in two common situations:

- an agent could upload the wrong document or keep an unusable OCR result, but had no clean way to delete it before it became part of the workflow;
- a dossier created too early or by mistake could still appear in the list even though it had not yet entered the real preparation workflow.

The goal of this day was therefore not to redesign the architecture, but to improve **workflow flexibility before business commitment**, while preserving **strict protection once a document or dossier enters a more advanced lifecycle state**.

The implementation followed two principles:

1. allow cleanup while the object is still in a removable preparation state;
2. block deletion once the object becomes part of the validated business workflow.

### 2.2 `DocumentController::destroy()` implementation

A new `destroy()` method was added to `DocumentController` to make document deletion safe and transactional.

The method follows this logic:

1. retrieve the authenticated user;
2. open a database transaction;
3. lock the document row with `lockForUpdate()`;
4. verify deletion authorization;
5. verify deletion eligibility by status;
6. verify that the document is not already attached to a dossier or rubrique;
7. delete the database row;
8. delete the stored file from disk after transaction success.

Implementation core:

```php
public function destroy(Request $request, Document $document): JsonResponse
{
    /** @var User $user */
    $user = $request->user();

    $result = DB::transaction(function () use ($document, $user) {
        /** @var Document $lockedDocument */
        $lockedDocument = Document::query()
            ->whereKey($document->id)
            ->lockForUpdate()
            ->firstOrFail();

        if (! $this->canDeleteDocument($user, $lockedDocument)) {
            return response()->json([
                'message' => 'You are not allowed to delete this document.',
            ], 403);
        }

        $currentStatus = $lockedDocument->status instanceof DocumentStatus
            ? $lockedDocument->status->value
            : (string) $lockedDocument->status;

        if (! in_array($currentStatus, [
            DocumentStatus::UPLOADED->value,
            DocumentStatus::FAILED->value,
            DocumentStatus::PROCESSED->value,
        ], true)) {
            return response()->json([
                'message' => 'Only UPLOADED, FAILED, or PROCESSED documents can be deleted.',
                'current_status' => $currentStatus,
            ], 422);
        }

        if ($lockedDocument->rubrique_id !== null || $lockedDocument->dossier_id !== null) {
            return response()->json([
                'message' => 'You cannot delete a document that is already attached to a dossier/rubrique.',
            ], 422);
        }

        $filePath = $lockedDocument->file_path;
        $lockedDocument->delete();

        return $filePath;
    });

    if ($result instanceof JsonResponse) {
        return $result;
    }

    if (is_string($result) && $result !== '') {
        Storage::disk('local')->delete($result);
    }

    return response()->json([
        'message' => 'Document deleted successfully.',
    ], 200);
}
```

### 2.3 Role-based deletion authorization

Deletion is intentionally not open to every authenticated role.

The authorization helper implements the following rule:

- `ADMIN` can delete any eligible document;
- `AGENT` can delete only their own eligible documents;
- `GESTIONNAIRE` cannot delete documents through this flow.

```php
private function canDeleteDocument(User $user, Document $document): bool
{
    if ($this->hasRole($user, UserRole::ADMIN)) {
        return true;
    }

    return $this->hasRole($user, UserRole::AGENT)
        && (int) $document->user_id === (int) $user->id;
}
```

**Business meaning:** the delete action is treated as a preparation-side cleanup capability, not a review-side capability.

---

### 2.4 Status-based deletion eligibility

To prevent unsafe removals, deletion is allowed only for the following technical states:

- `UPLOADED`
- `FAILED`
- `PROCESSED`

This means the document may still be removed before it becomes final in the technical workflow.

```php
if (! in_array($currentStatus, [
    DocumentStatus::UPLOADED->value,
    DocumentStatus::FAILED->value,
    DocumentStatus::PROCESSED->value,
], true)) {
    return response()->json([
        'message' => 'Only UPLOADED, FAILED, or PROCESSED documents can be deleted.',
        'current_status' => $currentStatus,
    ], 422);
}
```

Protected states therefore remain non-deletable, especially:

- `PROCESSING`
- `VALIDATED`

This keeps the workflow coherent and prevents deletion of documents already validated for business usage.

---

### 2.5 Attachment protection

A document already linked to a dossier or rubrique must not disappear from the workflow.

For that reason, the controller explicitly blocks deletion when the document is already attached:

```php
if ($lockedDocument->rubrique_id !== null || $lockedDocument->dossier_id !== null) {
    return response()->json([
        'message' => 'You cannot delete a document that is already attached to a dossier/rubrique.',
    ], 422);
}
```

**Result:** the delete flow remains compatible with the current business structure:

```text
Dossier
  └── Rubriques
        └── Documents
```

A document can be cleaned up before attachment, but not once it is already part of a dossier review context.

---

### 2.6 File-system cleanup

The implementation does not stop at deleting the database row.  
If the transaction completes successfully, the physical file is also removed from storage:

```php
if (is_string($result) && $result !== '') {
    Storage::disk('local')->delete($result);
}
```

This prevents orphaned local files and keeps the repository of uploaded documents technically clean.

## 3) Backend – Dossier Deletion Rule Tightening

### 3.1 `Dossier::canBeDeleted()` simplified

The dossier business rule was tightened at model level.

Before this change, dossier deletion logic was more permissive.  
Now the rule is explicit:

```php
public function canBeDeleted(): bool
{
    return $this->status === DossierStatus::RECEIVED;
}
```

**Meaning:** only dossiers that are still in the initial preparation state may be removed.

This prevents deletion once the dossier has already entered:

- `IN_PROGRESS`
- `TO_VALIDATE`
- `PROCESSED`
- `EXPORTED`

---

### 3.2 `DossierController::destroy()` enforcement

The controller was updated to expose the stricter rule clearly through the API.

```php
public function destroy(Request $request, Dossier $dossier): JsonResponse
{
    /** @var User $user */
    $user = $request->user();

    if (! $this->canManageOwnDossier($user, $dossier)) {
        return response()->json([
            'message' => 'You are not allowed to delete this dossier.',
        ], 403);
    }

    if (! $dossier->canBeDeleted()) {
        return response()->json([
            'message' => 'Only dossiers in RECEIVED status can be deleted.',
            'current_status' => $dossier->status->value,
        ], 422);
    }

    $dossier->delete();

    return response()->json([
        'message' => 'Dossier deleted successfully.',
    ], 200);
}
```

### 3.3 Business impact

This makes the dossier workflow more defendable:

- draft dossier not yet engaged in business preparation → removable;
- dossier already engaged in preparation/review/finalization → protected.

This change is more coherent with the reimbursement workflow and avoids erasing objects that already started carrying business meaning.

## 4) Frontend – Documents List Deletion Flow

### 4.1 Client-side eligibility filtering

`DocumentsList.jsx` was updated so that the delete action appears only when the document satisfies the same conditions enforced by the backend.

```javascript
const canDeleteDocument = (doc) => {
  const canDeleteByStatus = ['UPLOADED', 'FAILED', 'PROCESSED'].includes(doc.status);
  const isAttached = doc.rubrique_id != null || doc.dossier_id != null;

  if (!canDeleteByStatus || isAttached) {
    return false;
  }

  if (currentRole === 'ADMIN') {
    return true;
  }

  if (currentRole === 'AGENT') {
    return Number(doc.user_id) === Number(currentUser?.id);
  }

  return false;
};
```

This avoids showing deletion controls for documents that are clearly non-deletable.

---

### 4.2 Confirmation flow

Instead of deleting directly, the page now opens a confirmation modal.

```javascript
const requestDeleteDocument = (doc) => {
  if (!canDeleteDocument(doc) || isDeleteConfirming) {
    return;
  }

  setDeleteTargetDocument(doc);
};

const handleConfirmDelete = async () => {
  if (!deleteTargetDocument) {
    return;
  }

  const targetId = deleteTargetDocument.id;

  setError(null);
  setSuccessMessage(null);
  setIsDeleteConfirming(true);
  setDeletingDocumentId(targetId);

  try {
    const response = await api.delete(`/documents/${targetId}`);
    setSuccessMessage(response.data?.message || 'Document deleted successfully.');
    setDeleteTargetDocument(null);

    const nextPage = documents.length === 1 && currentPageRef.current > 1
      ? currentPageRef.current - 1
      : currentPageRef.current;

    setLoading(true);
    const docs = await fetchDocuments(nextPage);
    setupPolling(docs);
  } catch (err) {
    setError(getApiErrorMessage(err, 'Failed to delete document.'));
  } finally {
    setIsDeleteConfirming(false);
    setDeletingDocumentId(null);
  }
};
```

### 4.3 Action rendering refinement

The delete button is now appended only when allowed, while the main document action remains status-driven (`Validate`, `View`, `Retry`, `Processing...`, `Pending...`).

```javascript
const canDeleteDossier = (dossier) => {
```

This implements the intended behavior:

- `ADMIN` can delete any dossier still in `RECEIVED`;
- `AGENT` can delete only their own dossier still in `RECEIVED`;

### 6.1 Validation workflow kept intact

`DocumentValidationController.php` remains aligned with the technical workflow:

- only `PROCESSED` documents can be validated;
- validation still records corrections and creates a new extraction version;
- the document still becomes `VALIDATED` after successful human validation.

This means the new deletion flow does **not** weaken the technical validation pipeline.

No business redesign was introduced in dossier review.

The core review flow remains unchanged:

```text
RECEIVED -> IN_PROGRESS -> TO_VALIDATE -> PROCESSED -> EXPORTED
```

and document decisions still remain separate from the technical document lifecycle.

## 9) Architecture State After Day 23

The application now handles deletion with a more realistic business policy:

### Document side
- removable while still technical and not yet committed to dossier workflow;
- protected once attached or technically finalized.

### Dossier side
- removable only while still in the initial `RECEIVED` state;
- protected once actual preparation or review begins.

This reinforces a key design principle of the project:

- **flexibility before business commitment**
- **protection after workflow engagement**
