## 1) Objective of Day 25

Before Day 25, the AI microservice already worked correctly with the existing application workflow:

- upload from React
- persistence and orchestration in Laravel
- async processing through the queue
- OCR and extraction in FastAPI
- validation and review in Laravel/React

However, the OCR and extraction layer still had important limitations:

- OCR depended mainly on Tesseract
- confidence values were not sufficiently credible
- provider extraction was still too heuristic on some documents

Day 25 focused on improving the intelligence and robustness of the AI service while respecting the project architecture:

- **React communicates only with Laravel**
- **Laravel remains the orchestration layer and source of truth**
- **FastAPI remains stateless and limited to OCR / extraction / AI processing**

## 5) OCR Layer Upgrade

### 5.1 PaddleOCR integration

PaddleOCR was integrated as the main OCR engine inside `ai/main.py`.

Implemented behavior:

- initialize PaddleOCR once at startup
- preprocess image
- run PaddleOCR first
- normalize OCR output into internal line-based structures
- expose the actual engine used in `meta.ocr_engine`

### 5.2 Tesseract fallback retained

Tesseract was intentionally preserved as a fallback engine.

Fallback occurs when:

- PaddleOCR raises an exception
- PaddleOCR produces weak/unusable output
- the result does not pass the OCR usability checks

### 5.3 FastAPI execution safety

The endpoint remains synchronous (`def`, not `async def`) to avoid blocking FastAPI’s event loop with CPU/GPU-heavy OCR work.

This keeps the service operationally safer under concurrent requests.

## 6) GPU Acceleration for PaddleOCR

During testing, CPU PaddleOCR proved functionally correct but too slow for practical queue usage.

### 6.1 Problem observed

With CPU execution, PaddleOCR requests took too long and caused processing latency high enough to destabilize the queue workflow during real testing.

### 6.2 GPU enablement

The environment was upgraded to use the GPU build of PaddlePaddle.

Verification performed in the AI virtual environment:

- `paddle.device.is_compiled_with_cuda()` returned `True`
- `paddle.device.cuda.device_count()` returned `1`

### 6.3 Operational result

After GPU activation:

- PaddleOCR ran successfully on `gpu:0`
- request processing time dropped significantly
- the Laravel queue worker completed document jobs normally again

This transformed PaddleOCR from an experimental enhancement into a usable primary OCR engine in the current local environment.

## 7) spaCy Integration

### 7.1 Role of spaCy in the project

spaCy was added as a **post-processing support layer**, not as the main extractor.

Its role is to improve provider-name extraction after OCR by:

- detecting organization-like spans
- supporting provider validation
- helping reduce noisy provider extraction

### 7.2 Practical implementation

The Day 25 implementation uses:

- `fr_core_news_sm`
- `EntityRuler`
- lightweight provider-oriented patterns such as:
  - `pharmacie`
  - `laboratoire`
  - `clinique`
  - `cabinet`
  - `centre`
  - `compagnie`
  - `société`
  - `dr`
  - `docteur`

### 7.3 Scope deliberately controlled

spaCy is used to support:

- `provider_name`

It does **not** replace the extraction logic for:

- `invoice_date`
- `total_ttc`

Those remain primarily rule-based and normalization-based, which keeps the scope academically defendable and technically stable.

## 8) Extraction Logic Improvements

The extraction layer was strengthened while preserving the same required output fields.

### 8.1 `invoice_date`

Improved through:

- broader numeric date matching
- French textual date support
- normalization to `YYYY-MM-DD`
- contextual keyword support
- future-date penalty in confidence scoring

### 8.2 `total_ttc`

Improved through:

- stronger focus on `Total TTC` and similar final-payment keywords
- Tunisian amount support with 2 or 3 decimals
- support for `DT` and `TND`
- stronger discrimination against `TVA`, `HT`, and subtotal-like amounts
- preference for lower/final summary zone amounts

### 8.3 `provider_name`

Improved through:

- OCR top-zone heuristics
- spaCy/entity support
- provider-specific cleanup rules
- rejection of weak/noisy candidates
- specificity tuning to avoid generic results such as `CLINIQUE` when a richer candidate exists

## 9) Provider Specificity Refinement

After the first Day 25 evaluation, one real weakness remained visible:

- some documents could still produce generic provider outputs such as:
  - `CLINIQUE`
  - `PHARMACIE`
  - `LABORATOIRE`

To correct this, the provider ranking logic was refined.

### 9.1 Rules added

- penalty for generic single-token provider labels
- reward for richer multi-token provider names
- merged top-line candidate evaluation to capture split headers
- deduplication of equivalent provider candidates after normalization
- bonus for valid multiword names beginning with provider keywords

### 9.2 Result

Provider extraction became more specific and more aligned with real Tunisian medical documents and headers.

## 10) Confidence Scoring Refactor

Before Day 25, confidence values were still too close to decorative field defaults.

Day 25 replaced that with computed confidence.

### 10.1 Global OCR base

A global OCR base is computed from:

- OCR engine confidence
- normalized text quality score

### 10.2 Field-specific confidence

Each required field now uses its own extraction signals.

#### `invoice_date`
Signals considered:

- date matched
- date normalized successfully
- keyword context present
- OCR line quality
- future-date penalty

#### `total_ttc`
Signals considered:

- amount matched
- `Total TTC` / final-total keyword context
- currency detected (`DT`, `TND`)
- amount plausibility
- decimal quality
- OCR line quality

#### `provider_name`
Signals considered:

- provider candidate matched
- provider keyword context
- spaCy/entity support
- line cleanliness
- OCR line quality

### 10.3 Result

Confidence values became more meaningful and better aligned with extraction quality, which improves the human review layer in the validation UI.
