## 2) Frontend Refactor – `DossierDetail`

### 2.1 Migration to the new backend structure
The page was refactored so that it no longer assumes a simplified direct relation between dossier and documents. Instead, the detail page now works with the nested business structure:

```text
Dossier
  └── Rubriques
        └── Documents
```

A lightweight normalization layer was introduced to safely consume the real API response and handle:
- `dossier`
- `rubriques`
- nested `documents`
- `requested_total`
- `current_total`
- `display_total`

This prevents fragile rendering logic and makes the page more resilient to nullable arrays and nested payload differences.

### 2.2 Single source of truth strategy
`DossierDetail` was implemented as the single source of truth for dossier state. Instead of duplicating local state in several child sections, the page now follows a consistent pattern:

1. fetch dossier detail from backend
2. render from backend truth
3. perform mutation
4. refresh dossier detail
5. re-render from updated backend data

This approach was intentionally chosen over optimistic UI because one mutation may impact document decision status, rubrique status, dossier status, dossier totals, and available UI actions. For this Day 20 refactor, correctness and consistency were prioritized over micro-optimizations.

## 3) New Service Layer for Dossier Workflow

A dedicated service layer was consolidated in:

- `frontend/src/services/dossierWorkflow.js`

This service centralizes the dossier workflow API calls used by the page:
- `getDossierDetail(id)`
- `createRubrique(dossierId, payload)`
- `attachDocuments(rubriqueId, documentIds)`
- `detachDocument(rubriqueId, documentId)`
- `submitDossier(id)`
- `processDossier(id)`
- `acceptDocument(id, note)`
- `rejectDocument(id, note)`
- `rejectRubrique(rubriqueId, note)`
- `getValidatedDocuments()`

This keeps raw HTTP calls out of JSX and makes the page easier to read and evolve.

## 4) Dossier Detail Rendering Improvements

### 4.1 Dossier header
The dossier header now displays the key business information expected in the new workflow:
- `numero_dossier`
- `assured_identifier`
- dossier `status`
- `montant_total` (stored total)
- `requested_total`

- `episode_description`
- `notes`
- `created_at`
- `updated_at`

Formatting helpers were added to improve rendering stability for amounts, nullable values, dates, and final display total behavior.

### 4.2 Rubrique rendering
The page now renders a dedicated rubrique list, where each rubrique displays:
- title
- rubrique status
- notes
- number of contained documents

This makes the business grouping level visible in the frontend, which was the main conceptual gap remaining after the backend refactor of Day 19.

### 4.3 Document rendering inside each rubrique
Each rubrique now displays its own documents table with:
- document ID
- original filename
- technical document status
- business decision status
- extracted `total_ttc`
- decision note
- available actions depending on dossier state

This clearly separates the technical document lifecycle (`VALIDATED`, etc.) from the reimbursement decision lifecycle (`PENDING`, `ACCEPTED`, `REJECTED`).

## 5) Workflow Actions Added to the Browser

### 5.1 Create rubrique
A rubrique creation block was added directly to the dossier detail page.

Implemented behavior:
- title required, optional notes
- pending button state during request

- automatic refresh of dossier detail after success

### 5.2 Attach validated documents
An attach-documents modal was added at rubrique level.

Implemented behavior:
- load validated documents from the backend
- fetch paginated document data until all pages are collected

- exclude already attached documents
- allow multi-selection
- close modal after success and refresh dossier detail

### 5.3 Detach document
A detach action was added for editable dossier states.

Implemented behavior:
- user confirmation before detach
- pending state on the target document
- automatic refresh after success
- requested total updates correctly after removal

### 5.4 Submit dossier
A submit button was added to transition the dossier from preparation phase to review phase.

Implemented behavior:
- confirmation prompt and pending state
- success and error feedback
- automatic refresh after success
- transition to `TO_VALIDATE`

### 5.5 Accept document
A document acceptance action was added for dossier review.

Implemented behavior:
- available only in review phase (`TO_VALIDATE`)
- `decision_status` changes to `ACCEPTED`
- `current_total` updates correctly based on accepted documents
- automatic refresh after success

### 5.6 Reject document
A document rejection modal was added.

Implemented behavior:
- rejection note required in the UI
- note persisted and displayed in the table
- automatic refresh after success
- improved traceability and auditability of rejection actions

### 5.7 Reject rubrique
A dedicated rubrique rejection modal was added to replace the initial native dialog approach.

Implemented behavior:
- optional decision note

- automatic refresh after success

### 5.8 Process dossier
A process button was added to finalize the dossier.

Implemented behavior:
- dossier status changes to `PROCESSED`

- automatic refresh after success

## 6) Status-Driven UI Control

### 6.1 Dossier-state action matrix
The page was updated to show or hide actions according to the dossier business status:
- **RECEIVED** → rubrique creation
- **IN_PROGRESS** → create rubrique, attach documents, detach documents, submit dossier
- **TO_VALIDATE** → accept/reject documents, reject rubrique, process dossier
- **PROCESSED / EXPORTED** → read-only behavior

### 6.2 Frozen dossier protection
When the dossier reaches `PROCESSED` or `EXPORTED`, the page now displays a clear finalized banner and disables the mutation flow visually.

---

## 7) UX and Stability Improvements

- **Success and error feedback:** visible feedback is displayed for every mutation, and no action fails silently.
- **Auto-dismiss success messages:** transient success messages now disappear automatically after a short delay.
- **Modal consistency:** rubrique rejection was upgraded from native browser dialogs to a real React modal.
- **Pending state protection:** buttons are disabled during requests to prevent duplicate clicks.
