# 1) Objective of Day 38

After Day 37, the project already had:

- a stabilized dossier workflow;
- full EN/FR localization on the frontend;
- a persistent workflow notification subsystem;
- a professional dossier-detail page with current-state cards for escalation and return-to-preparation context.

However, one major traceability gap still remained:

- the application showed the **current dossier state**;
- the notification table captured **recipient-oriented workflow alerts**;
- but there was still no dedicated **append-only business journal** for the full dossier lifecycle itself.

This meant the system could explain the current state, but not fully reconstruct the dossier’s real chronological history for cases involving:

- multiple returns to preparation;
- repeated resubmissions;
- repeated escalations;
- supervisor returns to claims manager;
- complement cycles;
- final processing.

Day 38 therefore introduced a true workflow-history layer whose purpose is not notification delivery, but **immutable traceability**.

The implementation was intentionally split into two responsibilities:

- `dossiers` table keeps the **current snapshot** of the dossier;
- `dossier_workflow_events` keeps the **append-only history** of what happened over time.

This separation is one of the most important architectural gains of the day.

# 2) Architecture and Scope Decisions

## 2.1 Dedicated workflow-history table instead of reusing notifications

A key design decision was made early and kept until the end:

> `app_notifications` must not be reused as the dossier traceability source.

The notification subsystem remains useful for:

- recipient-based alerts;
- unread/read behavior;
- action links;
- cross-role operational feedback.

But it is **not** the correct persistence model for a dossier audit journal because it is centered around:

- a recipient user;
- notification read state;
- delivery semantics.

The new workflow-event table is instead centered around:

- the dossier;
- the business event type;
- state transition information;
- note preservation;
- immutable chronology.

That makes the traceability layer much cleaner and easier to defend academically.

## 2.2 Append-only contract made explicit in schema and model

The new history layer was intentionally designed as append-only.

This was made visible at multiple levels:

- no `updated_at` column in the migration;
- `created_at` only;
- `public $timestamps = false` in the model;
- dedicated `record(...)` service methods that only insert rows.

This avoids the ambiguity of a generic mutable Eloquent model and makes the audit intent explicit.

## 2.3 After-commit best-effort emission

Another important design rule was preserved from the Day 36 notification subsystem:

- workflow transition first;
- side effects second.

Both notifications and workflow-history rows are emitted **after commit**.

That means:

- if the transition fails, no history row is written;
- if history insert fails, the business transition still succeeds;
- history logging never rolls back the dossier workflow.

This was the correct production-style trade-off for a late-stage PFE.

## 2.4 Current context and full history remain separate in the UI

The frontend deliberately keeps three distinct concepts inside `DossierDetail`:

1. current hierarchical context;
2. current return-to-preparation context;
3. full immutable workflow history.

This means the upper cards remain **current-state explanation blocks**, while the new history card becomes the only place where the **full multi-cycle chronology** is shown.

That prevents duplication and keeps the page readable.

# 3) Database Foundation – `dossier_workflow_events`

A new table named `dossier_workflow_events` was introduced as the dedicated persistence layer for dossier traceability.

Representative structure:

```php
Schema::create('dossier_workflow_events', function (Blueprint $table) {
    $table->id();

    $table->foreignId('dossier_id')->constrained('dossiers')->cascadeOnDelete();
    $table->foreignId('actor_id')->nullable()->constrained('users')->nullOnDelete();

    $table->string('event_type', 100);
    $table->string('from_status', 100)->nullable();
    $table->string('to_status', 100)->nullable();
    $table->string('title', 255);
    $table->text('description')->nullable();
    $table->text('note')->nullable();
    $table->json('meta')->nullable();

    $table->timestamp('created_at')->useCurrent();

    $table->index(['dossier_id', 'created_at']);
    $table->index('event_type');
});
```

### Why this schema matters

This design gives the project a real business-history structure:

- `dossier_id` anchors the row to the case file;
- `actor_id` preserves who triggered the action when available;
- `event_type` makes the event machine-readable;
- `from_status` and `to_status` preserve workflow transition semantics;
- `title` and `description` support direct frontend rendering;
- `note` preserves operator-provided reason fields;
- `meta` allows lightweight contextual extension;
- `created_at` is the only timestamp needed for the immutable journal.

This is much cleaner than trying to infer traceability later from scattered snapshot fields.

# 4) Backend Model and Service Layer

## 4.1 `DossierWorkflowEvent` model

A dedicated model was introduced for the new table.

The model intentionally reflects the append-only nature of the journal:

- `$timestamps = false`;
- `created_at` cast to `datetime`;
- `meta` cast to `array`;
- relation to `dossier()`;
- relation to `actor()`.

Representative structure:

```php
class DossierWorkflowEvent extends Model
{
    public $timestamps = false;

    protected $fillable = [
        'dossier_id',
        'actor_id',
        'event_type',
        'from_status',
        'to_status',
        'title',
        'description',
        'note',
        'meta',
        'created_at',
    ];

    protected $casts = [
        'meta' => 'array',
        'created_at' => 'datetime',
    ];
}
```

## 4.2 Centralized `DossierWorkflowEventService`

Instead of writing rows directly inside controllers, the implementation introduced `DossierWorkflowEventService`.

This service is responsible for:

- creating one immutable row per business transition;
- preserving a minimal stable metadata shape;
- normalizing enum values safely;
- logging failures without breaking the main workflow.

Representative service pattern:

```php
public function record(
    Dossier $dossier,
    ?User $actor,
    string $eventType,
    ?string $fromStatus,
    ?string $toStatus,
    string $title,
    ?string $description = null,
    ?string $note = null,
    array $meta = []
): void {
    try {
        DossierWorkflowEvent::create([
            'dossier_id' => $dossier->id,
            'actor_id' => $actor?->id,
            'event_type' => $eventType,
            'from_status' => $fromStatus,
            'to_status' => $toStatus,
            'title' => $title,
            'description' => $description,
            'note' => $note,
            'meta' => [
                'numero_dossier' => $dossier->numero_dossier,
                'actor_role' => $this->normalizeEnumValue($actor?->role),
                ...$meta,
            ],
            'created_at' => now(),
        ]);
    } catch (\Throwable $e) {
        Log::error('DossierWorkflowEventService: failed to record workflow event.', [
            'dossier_id' => $dossier->id,
            'event_type' => $eventType,
            'error' => $e->getMessage(),
        ]);
    }
}
```

### Why this matters

This keeps string construction and history mechanics out of controllers.
Controllers remain focused on:

- authorization;
- state mutation;
- response payloads.

The service owns:

- event semantics;
- note persistence;
- immutable insertion.

## 4.3 Thin business-event helpers

The service also exposes thin wrappers for the main transitions, including:

- `recordCreation(...)`
- `recordSubmission(...)`
- `recordResubmission(...)`
- `recordReturnToPreparation(...)`
- `recordEscalation(...)`
- `recordSupervisorApproval(...)`
- `recordSupervisorReturn(...)`
- `recordSupervisorComplement(...)`
- `recordProcessing(...)`

This makes the event-writing code readable at controller call sites while keeping the event language centralized.

# 5) Backend Workflow Wiring

The main technical value of Day 38 came from wiring the existing dossier workflow into the new immutable event journal.

## 5.1 `DossierController`

The controller was extended in four major workflow areas.

### Creation

When a dossier is created, the system now records:

- `DOSSIER_CREATED`

### Submission and resubmission

When a dossier enters review, the controller now distinguishes:

- first submission → `DOSSIER_SUBMITTED_FOR_REVIEW`
- reopened dossier submission → `DOSSIER_RESUBMITTED_FOR_REVIEW`

The crucial implementation detail is that the old state is captured **before** mutation:

```php
$fromStatus = $lockedDossier->status->value;
$wasResubmission = $lockedDossier->status === DossierStatus::AWAITING_COMPLEMENT;
```

### Return to preparation

When a claims manager reopens preparation, the controller now records:

- `DOSSIER_RETURNED_TO_PREPARATION`

and preserves the return note in the history row.

### Processing

When a dossier is finalized, the controller now records:

- `DOSSIER_PROCESSED`

## 5.2 `DossierEscalationController`

The escalation controller was also wired to the history service.

### Escalation

When a claims manager escalates a dossier, the system records:

- `DOSSIER_ESCALATED`

with the escalation reason preserved as the event note.

### Supervisor approval

When the supervisor approves escalation, the system records:

- `SUPERVISOR_APPROVED_ESCALATION`

### Supervisor return to claims manager

When the supervisor returns the dossier for additional review, the system records:

- `SUPERVISOR_RETURNED_TO_CLAIMS_MANAGER`

### Supervisor complement request

When the supervisor requests complement, the system records:

- `SUPERVISOR_REQUESTED_COMPLEMENT`

with the complement note preserved in the history row.

## 5.3 After-commit pattern preserved

All history writing follows the same safe pattern:

1. capture `fromStatus` before the transition;
2. perform the transition inside `DB::transaction(...)`;
3. emit notification and/or workflow-history row inside `DB::afterCommit(...)`.

This keeps the write-side logic technically consistent with Day 36.

# 6) Read API and Model Relation

## 6.1 `Dossier::workflowEvents()` relation

To expose the history cleanly, the dossier model was extended with:

```php
public function workflowEvents(): HasMany
{
    return $this->hasMany(DossierWorkflowEvent::class)
        ->orderBy('created_at', 'asc');
}
```

This makes the read-side API explicit and keeps the history query anchored to the dossier aggregate.

## 6.2 New read endpoint

A new route was added under the dossier API group:

```php
Route::get('/{dossier}/workflow-events', [DossierController::class, 'workflowEvents']);
```

The corresponding controller method:

- checks dossier visibility with the existing `canViewDossier(...)` rule;
- loads workflow events with actor data;
- returns the events in chronological order;
- exposes the actor id/name/role, title, description, note, transition fields, metadata, and timestamp.

Representative response mapping:

```php
$events = $dossier->workflowEvents()
    ->with('actor')
    ->orderBy('created_at', 'asc')
    ->get()
    ->map(fn ($event) => [
        'id' => $event->id,
        'event_type' => $event->event_type,
        'from_status' => $event->from_status,
        'to_status' => $event->to_status,
        'title' => $event->title,
        'description' => $event->description,
        'note' => $event->note,
        'meta' => $event->meta,
        'created_at' => $event->created_at,
        'actor' => $event->actor ? [
            'id' => $event->actor->id,
            'name' => $event->actor->name,
            'role' => $event->actor->role instanceof \BackedEnum
                ? $event->actor->role->value
                : $event->actor->role,
        ] : null,
    ]);
```

This endpoint is the bridge that allows the frontend to render the immutable journal without coupling itself to the database schema.

## 9.3 Stale-history fix after successful mutations

One real issue appeared after the first frontend pass:

- the history loaded correctly on mount;
- but successful actions still only refreshed the dossier detail snapshot;
- the new history card could therefore stay stale until manual reload.

This was fixed through a unified helper that refreshes both:

- dossier detail;
- workflow history.

Representative helper pattern:

```jsx
const refreshDossierView = async () => {
  await refreshDetailSilently();
  await refreshWorkflowHistory();
};
```

This helper was then wired into all relevant success paths, including:

- submit dossier;
- process dossier;
- escalate dossier;
- supervisor approval;
- supervisor return;
- supervisor complement request;
- return to preparation.

This was critical to make the new timeline trustworthy in real use.

# 16) Main Technical Gain of Day 38

The biggest gain of Day 38 is not simply the addition of another card in the dossier detail page.

The real gain is that the project now distinguishes clearly between three different concepts that enterprise workflow systems often confuse:

- current snapshot state;
- recipient-based notifications;
- immutable business traceability.

By separating these properly, the project becomes much stronger technically and academically.

It is now easier to defend that the system:

- preserves current dossier context cleanly;
- notifies the right actors at the right moments;
- records the full lifecycle independently of notification delivery;
- supports repeated review cycles without losing earlier history.

That makes Day 38 one of the most important architectural consolidation days of the whole implementation phase.
