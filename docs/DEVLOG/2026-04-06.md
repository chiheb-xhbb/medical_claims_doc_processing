## 2) Main Business Decision of Day 27

A key design decision was locked before implementation:

### Escalation does **not** happen after `PROCESSED`

The dossier cannot be escalated after final processing because `PROCESSED` is already the frozen terminal state in the current model.

So the hierarchical workflow was designed to branch from:

```text
TO_VALIDATE ‚Üí EN_DEROGATION
```

and not from:

```text
PROCESSED ‚Üí ...
```

### Final state remains unique

Another important decision was preserved:

- `PROCESSED` remains the **only final dossier state**
- no new terminal state such as `VALIDE` was introduced

This keeps the dossier state machine academically cleaner and avoids state duplication.

## 4) Backend ‚Äì Enumeration Extension

### 4.1 `UserRole` extended

The role enum was updated to add:

- `CHEF_HIERARCHIQUE`

This makes the hierarchical reviewer an explicit domain actor inside the backend instead of an informal concept handled by comments or ad-hoc conditions.

### 4.2 `DossierStatus` extended

The dossier enum was extended with two new states:

- `EN_DEROGATION`
- `COMPLEMENT_ATTENDU`

The final dossier lifecycle therefore becomes:

- `RECEIVED`
- `IN_PROGRESS`
- `TO_VALIDATE`
- `EN_DEROGATION`
- `COMPLEMENT_ATTENDU`
- `PROCESSED`

These two new states were chosen carefully:

- `EN_DEROGATION` = dossier currently under hierarchical arbitration
- `COMPLEMENT_ATTENDU` = dossier suspended pending missing information/documents

No other enum was changed for Day 27:

- `DocumentStatus` remains technical
- `DocumentDecisionStatus` remains dossier-review level
- `RubriqueStatus` remains rubrique-review level

## 5) Database ‚Äì Dossier Escalation Traceability

A new migration was created to extend the `dossiers` table with hierarchical review traceability fields.

### Added columns

- `escalated_by`
- `escalated_at`
- `escalation_reason`
- `chef_decision_type`
- `chef_decision_by`
- `chef_decision_at`
- `chef_decision_note`

### Purpose of these fields

These columns make the escalation flow auditable without creating a whole new table at this stage.

They allow the application to store:

- who escalated the dossier
- when it was escalated
- why it was escalated
- what the Chef decided
- who took the hierarchical decision
- when that decision was taken
- the decision note associated with the action

### Design choice

A snapshot-style traceability design was chosen for Day 27 instead of a full `dossier_escalations` history table.

This was intentional:

- enough for a first clean implementation
- consistent with the existing action-based fields already used in the project
- lower risk than introducing a new entity in the middle of the sprint

## 9) Hierarchical Business Rules Implemented

### 9.1 Escalate dossier

A Gestionnaire (or Admin) can escalate a dossier only when:

- dossier is in `TO_VALIDATE`
- dossier is not frozen
- all document decisions are already completed

If valid:

- dossier becomes `EN_DEROGATION`
- escalation traceability is recorded
- previous Chef decision snapshot is cleared for the new cycle

### 9.2 Approve derogation

A Chef Hi√©rarchique (or Admin) can approve only when:

- dossier is in `EN_DEROGATION`
- dossier is not frozen
- all document decisions are already completed

If valid:

- dossier becomes `PROCESSED`
- final total is frozen from current accepted amount
- `processed_by` / `processed_at` are filled
- Chef decision metadata is stored as `APPROVED`

### 9.3 Return to Gestionnaire

A Chef Hi√©rarchique can return a dossier only when:

- dossier is in `EN_DEROGATION`
- note is mandatory

If valid:

- dossier returns to `TO_VALIDATE`
- Chef decision metadata is stored as `RETURNED`

### 9.4 Request complement

A Chef Hi√©rarchique can request complement only when:

- dossier is in `EN_DEROGATION`
- note is mandatory

If valid:

- dossier becomes `COMPLEMENT_ATTENDU`
- Chef decision metadata is stored as `COMPLEMENT_REQUESTED`

## 10) Important Audit Rule Preserved

A deliberate decision was made regarding document decisions when a dossier is returned to Gestionnaire.

### Chosen behavior

When Chef returns the dossier:

- existing document decisions are **kept**
- they are **not reset to `PENDING`**

### Why

Resetting all decisions would:

- destroy useful review work
- weaken auditability
- force the Gestionnaire to start from zero even if only one issue was raised

Keeping the existing decisions is therefore more realistic and more defensible architecturally.
