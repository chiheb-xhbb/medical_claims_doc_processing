# 1) Objective of Day 34

After Day 33, the password-reset flow was complete and the project had already reached a stable state for:

- secure document upload and OCR processing
- protected source-document access
- dossier / rubrique / document workflow
- escalation and return-to-preparation behavior
- account-management and password-reset flows

However, the final supervisor scope still contained important inconsistencies between the intended business direction and the implemented code.

The core problem was not a single bug, but a set of role-alignment gaps:

- the supervisor could not fully operate on dossiers they created themselves in preparation
- rubrique creation and attachment logic still contained older ownership restrictions
- document visibility and validation behavior remained partially inconsistent across backend and frontend
- some UI defaults still reflected an older escalation-only supervisor perspective
- the dossier-detail attach-documents modal still behaved as if validated documents had to belong to the connected user in all cases

Day 34 therefore became a **role-alignment and stabilization day**, focused on making the supervisor role operationally coherent in the real browser workflow.

# 2) Main Scope Decision of Day 34

A deliberate scope decision was kept throughout the work:

## 2.1 Notifications deferred again

Even though Sprint 4 planning still mentions a future `notifications` table and internal notification base, Day 34 did **not** implement that subsystem.

This was the correct decision because the urgent and directly testable gap was role-behavior consistency, not the creation of a new cross-cutting notification layer.

## 2.2 Supervisor scope widened through the existing architecture

The work deliberately reused the current application structure:

- React + Vite + Bootstrap on the frontend
- Laravel as the only orchestration and authorization layer
- MySQL as persistence
- FastAPI remaining strictly limited to OCR / extraction

No new architectural path was introduced.

The supervisor behavior was widened by correcting:

- Laravel controller authorization logic
- request-level role authorization
- frontend route protection
- page-level action visibility
- modal eligibility filtering

rather than by creating a special isolated supervisor subsystem.

# 3) Final Day 34 Business Direction Implemented

Day 34 effectively locked the following practical behavior for the supervisor role.

## 3.1 Preparation-side ownership

If the supervisor creates a dossier himself, he can now:

- create the dossier
- prepare it in `RECEIVED`, `IN_PROGRESS`, and `AWAITING_COMPLEMENT`
- add rubriques/sections
- upload documents
- attach and detach validated documents during preparation
- submit the dossier into the normal review flow

This gives the supervisor a real preparation-side operating capacity instead of leaving the role artificially limited to escalation screens only.

## 3.2 Submitted dossiers remain distinct from escalation review

Once a dossier enters normal submitted review, the normal first-pass reviewer remains separated from preparation ownership.

Day 34 did **not** redesign the whole dossier review chain again.

Instead, the work concentrated on making the supervisor able to:

- see the relevant dossiers and documents
- inspect and validate documents from the dedicated documents workspace
- continue to use escalation features where relevant

while preserving the rest of the already-stabilized dossier workflow behavior.

## 3.3 Documents workspace widened decisively

A key business clarification was adopted late in the day:

> The supervisor must be able to see all documents in the Documents section and validate processed documents like the claims manager.

This changed the previously narrower interpretation of the supervisor role in the documents area.

The final code therefore allows the supervisor to:

- list all documents in the documents workspace
- preview and download all source files
- validate processed documents
- view validated documents in read-only mode afterward

This became one of the most important Day 34 policy changes.

# 4) Backend – Dossier Visibility and Preparation Ownership

## 4.1 `DossierController` widened for supervisor preparation ownership

The dossier controller was updated so the supervisor can now participate correctly in preparation-side operations.

### Main effects

- supervisor can prepare dossiers they created themselves
- supervisor can manage own preparation dossiers in:
  - `RECEIVED`
  - `IN_PROGRESS`
  - `AWAITING_COMPLEMENT`
- supervisor can still see submitted dossiers relevant to supervision
- supervisor does not gain unlimited preparation visibility over everyone else’s unfinished dossiers

### Why this mattered

Before this correction, the frontend and some UX paths suggested supervisor preparation ownership, but the backend still rejected certain operations because the authorization helpers remained too narrow.

Day 34 fixed that drift at the controller level.

# 5) Backend – Document Visibility and File Access

## 5.1 `DocumentController` expanded the supervisor’s document workspace

The document controller became the main place where the supervisor scope changed materially.

### Final behavior

- `AGENT` still sees only own documents
- `CLAIMS_MANAGER` can see all documents
- `SUPERVISOR` can now see all documents
- `ADMIN` can see all documents

This applies both to:

- list visibility (`index()`)
- single document access (`show()`)
- file preview and download (`view()` / `download()`)

### Result

The documents workspace is no longer artificially limited for the supervisor.

This solved several operational problems:

- inability to inspect processed documents outside escalation context
- inability to verify validated documents centrally
- mismatch between role expectations and actual browser access

## 5.2 Delete and retry rules preserved with role-aware constraints

The controller also kept the existing guardrails around destructive actions.

Deletion still remains constrained by:

- allowed statuses only
- prohibition on attached documents

At the end of the day, supervisor delete behavior was aligned with claims manager in the documents area under the same status/attachment constraints, while retry behavior stayed role-aware and intentionally narrower.

# 6) Backend – Document Validation Authorization

## 6.1 `DocumentValidationController` aligned the final validation rule

A major late clarification from the business side changed the intended result:

- the supervisor should not be merely read-only in the document validation screen
- the supervisor should be able to validate processed documents like the claims manager

The validation controller was therefore updated so:

- `CLAIMS_MANAGER` can validate
- `SUPERVISOR` can validate
- `ADMIN` can validate
- `AGENT` remains owner-scoped for validation

### Important preserved rule

Validation still respects document lifecycle semantics:

- only `PROCESSED` documents can be validated
- `VALIDATED` documents remain read-only afterward

This means Day 34 widened **who can validate**, without breaking the technical document lifecycle.

# 7) Backend – Rubrique Preparation and Attachment Alignment

## 7.1 `RubriqueController` widened preparation ownership

A concrete bug appeared when the supervisor created a dossier and then attempted to create a rubrique/section:

> “You are not allowed to add rubriques to this dossier.”

The cause was simple: the rubrique controller still used older preparation helpers that did not include `SUPERVISOR`.

### Fix applied

The same preparation logic already used in `DossierController` was aligned into `RubriqueController`.

### Result

The supervisor can now:

- create sections on dossiers he created himself
- update or delete those sections during preparation
- attach and detach documents within the allowed preparation states

## 7.2 Attach-documents modal/backend mismatch fixed

Another important bug appeared later:

- the attach-documents modal reported that no validated documents were available
- yet validated documents actually existed
- the same issue affected both claims manager and supervisor

### Root cause

There was a frontend/backend mismatch:

- frontend modal filtering still owner-scoped too broadly
- backend `attachDocuments()` still blocked non-admin attachment when the document belonged to another uploader

### Final rule implemented

For rubrique attachment during preparation:

- `AGENT` remains owner-scoped
- `CLAIMS_MANAGER` can attach validated unassigned documents
- `SUPERVISOR` can attach validated unassigned documents
- `ADMIN` can attach validated unassigned documents

### Result

The attachment modal now reflects the real eligible document pool, and the backend authorization finally matches the UI.

# 11) Frontend – `DocumentValidation.jsx` Permission Cleanup

The document validation page was one of the most sensitive files of the day.

## 11.1 Old issue

The page still computed validation eligibility too narrowly for the supervisor and turned the screen into a permission-based read-only mode.

This produced the message:

> “You do not have permission to validate this document.”

for cases that the final business rule now explicitly allows.

## 11.2 Final behavior

The page now behaves as follows:

- `CLAIMS_MANAGER` can validate processed documents
- `SUPERVISOR` can validate processed documents
- `ADMIN` can validate processed documents
- `AGENT` can validate own processed documents
- only already-validated documents are read-only

### Result

The validation page now matches the backend authorization rule instead of enforcing an outdated frontend-only restriction.

- React still communicates only with Laravel

Day 34 is therefore a **workflow/RBAC consolidation day**, not an architectural redesign day.
