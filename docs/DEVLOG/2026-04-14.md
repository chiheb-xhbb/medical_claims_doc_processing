# 1) Objective of Day 32

After Day 31, the platform already had:

- secure document upload and protected file access
- OCR/extraction + validation
- dossier preparation and section-level organization
- escalation workflow already usable end-to-end
- account-management features and stabilized modal behavior

However, several business clarifications made the existing review logic no longer precise enough:

- the **Claims Manager** performs the first real review in normal `UNDER_REVIEW`
- the **Supervisor** should intervene only after escalation
- during escalation, the **Supervisor** must be able to override document decisions
- if the Supervisor returns the dossier, the Claims Manager must **not** reopen document-level decisions again
- the Claims Manager must also be able to return a dossier to preparation without escalating it
- when the dossier is reopened in preparation, the UI must show the **correct current reopen reason**, not stale snapshot data from an older action

Day 32 therefore became a **workflow refactor day**, not a design day.

The implementation targeted five concrete outcomes:

- add a real **Claims Manager → Return to Preparation** action
- tighten document/rubrique decision permissions by workflow phase
- prevent supervisor visibility from leaking into preparation states
- stabilize escalation behavior after a controller regression
- introduce a clean **canonical reopened-preparation context** so the top banner always explains the current reopen source correctly

# 2) Main Business Rules Frozen on Day 32

Before implementation, the workflow truth had to be clarified and locked.

## 2.1 Claims Manager performs the first review

Normal dossier review now follows:

```text
RECEIVED
   ↓
IN_PROGRESS
   ↓
UNDER_REVIEW
```

In this normal `UNDER_REVIEW` state:

- Claims Manager reviews documents
- Claims Manager may process the dossier
- Claims Manager may escalate the dossier
- Claims Manager may return the dossier to preparation

## 2.2 Supervisor only acts after escalation

The Supervisor is now a **strict escalation actor**, not a preparation actor.

So the Supervisor acts only when the dossier is in:

- `IN_ESCALATION`

and can then:

- approve escalation
- return to Claims Manager
- request complement
- modify document decisions during the escalation phase

## 2.3 Returned-from-supervisor is not a full review reset

When the Supervisor returns a dossier:

- dossier status goes back to `UNDER_REVIEW`
- old document decisions remain visible
- Claims Manager **cannot** accept/reject documents again
- Claims Manager can only:
  - process the dossier
  - escalate again

This preserves the first-pass review and avoids turning the returned dossier into a fake fresh review cycle.

## 2.4 Return to preparation and supervisor complement request are distinct actions

Both actions can reopen work in preparation, but they are not the same thing.

### Claims Manager return to preparation
- actor: Claims Manager/Admin
- source: first-level review
- reason: preparation correction before escalation

### Supervisor complement request
- actor: Supervisor/Admin
- source: escalation review
- reason: additional preparation requested during supervisor intervention

This distinction became important later in the day when the reopened-preparation banner exposed stale note data.

# 3) Final Workflow Implemented

The Day 32 implementation produces the following dossier review model.

## 3.1 Normal review path

```text
RECEIVED
   ↓
IN_PROGRESS
   ↓
UNDER_REVIEW
   ├── process dossier → PROCESSED
   ├── escalate dossier → IN_ESCALATION
   └── return to preparation → AWAITING_COMPLEMENT
```

## 3.2 Escalation review path

```text
IN_ESCALATION
   ├── approve escalation → PROCESSED
   ├── return to Claims Manager → UNDER_REVIEW
   └── request complement → AWAITING_COMPLEMENT
```

## 3.3 Reopened preparation path

```text
AWAITING_COMPLEMENT
   ↓
preparation owner updates dossier / sections / documents
   ↓
submit dossier
   ↓
UNDER_REVIEW
```

## 3.4 Returned-from-supervisor restriction

```text
IN_ESCALATION
   ↓
Supervisor returns dossier
   ↓
UNDER_REVIEW + chef_decision_type = RETURNED
```

In this special `UNDER_REVIEW` state:

- previous document decisions remain visible
- Claims Manager cannot modify document decisions
- Claims Manager can process the dossier or escalate again

This became the core semantic correction of Day 32.

# 4) Backend – Claims Manager Return to Preparation

## 4.1 New request validation class

A dedicated request class was created:

- `ClaimsManagerReturnRequest.php`

This validates:

- `return_note` as a required field
- note length/format rules
- actor authorization limited to:
  - `CLAIMS_MANAGER`
  - `ADMIN`

This keeps the workflow note validation out of the controller and follows the request-based pattern already used elsewhere.

## 4.2 First migration: return-to-preparation traceability

A dedicated migration was added:

- `2026_04_14_232921_add_return_to_preparation_fields_to_dossiers_table.php`

New fields introduced on `dossiers`:

- `returned_to_preparation_by`
- `returned_to_preparation_at`
- `returned_to_preparation_note`

The Claims Manager return action should not reuse generic dossier notes or supervisor decision fields, so the schema now stores it explicitly.

## 4.3 `Dossier` model extended

`backend/app/Models/Dossier.php` was updated with:

- new fillable return-to-preparation fields
- `returned_to_preparation_at` cast
- new relation:
  - `returnedToPreparationBy()`

## 4.4 `DossierController::returnToPreparation()` added

A new dossier workflow method was added:

- `returnToPreparation(ClaimsManagerReturnRequest $request, Dossier $dossier)`

### Behavior implemented

The action:

- is available to Claims Manager/Admin
- locks the dossier row with `lockForUpdate()`
- requires dossier status = `UNDER_REVIEW`
- rejects frozen dossiers
- blocks the action when the dossier is already in returned-from-supervisor mode
- updates the dossier to `AWAITING_COMPLEMENT`
- stores actor, timestamp, and return note

This belongs in `DossierController` because it is a normal dossier workflow action, not an escalation-controller action.

# 5) Backend – Supervisor Visibility and Review Authorization

## 5.1 Supervisor visibility corrected

The dossier list/detail visibility logic for `SUPERVISOR` was tightened.

The Supervisor should not keep seeing dossiers in `AWAITING_COMPLEMENT` as if they were still under supervisor work.

So supervisor visibility was reduced to:

- `IN_ESCALATION`
- `PROCESSED`

and no longer treats `AWAITING_COMPLEMENT` as an active supervisor state.

## 5.2 `DocumentDecisionController` aligned to the real workflow

The document decision controller was refactored so decision permissions now depend on:

- dossier status
- role
- and, in normal review, current decision state

### Final logic

#### In `IN_ESCALATION`
- `SUPERVISOR` may modify document decisions
- `ADMIN` may also do so

#### In normal `UNDER_REVIEW`
- `CLAIMS_MANAGER` may decide only `PENDING` documents
- already-decided documents are not re-opened
- if the dossier is returned from supervisor:
  - document decisions are completely locked

## 5.3 `RubriqueController::rejectAll()` aligned too

The whole-section rejection flow was updated to follow the same principles:

- normal review = Claims Manager/Admin
- escalation review = Supervisor/Admin
- returned-from-supervisor = locked for Claims Manager
- decision workflow allowed only in:
  - `UNDER_REVIEW`
  - `IN_ESCALATION`

## 5.4 `DocumentController` file access consistency fix

A smaller but important consistency fix was applied in `DocumentController`.

Previously, supervisor file access still allowed:

- `IN_ESCALATION`
- `AWAITING_COMPLEMENT`
- `PROCESSED`

This was corrected so the supervisor can access dossier-linked files only for:

- `IN_ESCALATION`
- `PROCESSED`

# 6) Backend – `DossierEscalationController` Recovery and Stabilization

During implementation, a severe regression appeared.

## 6.1 Error encountered

Escalation started crashing with a fatal PHP error:

```text
Cannot redeclare class App\Http\Controllers\Api\DocumentDecisionController
```

## 6.2 Root cause

`DossierEscalationController.php` had accidentally been overwritten with a copy of `DocumentDecisionController.php`.

So the file declared the wrong class:

- `DocumentDecisionController`
- instead of `DossierEscalationController`

This created a class-collision failure during Laravel boot.

## 6.3 Resolution

The controller was fully restored as the real escalation workflow controller, with methods:

- `escalate()`
- `approveDerogation()`
- `returnToGestionnaire()`
- `requestComplement()`

After restoration:

- Laravel caches/autoload were cleared
- escalation behavior returned to normal
- the workflow became stable again

# 10) Mid-Day Architecture Issue Discovered: Stale Reopen Note

A real modeling issue appeared after the first implementation.

## 10.1 Problem observed

If a Claims Manager first returned the dossier to preparation, then later the dossier was escalated and the Supervisor requested complement, the top banner still showed the **older Claims Manager return note** instead of the latest supervisor reopen reason.

So the UI displayed stale state even though the latest action was different.

## 10.2 Root cause

The backend was mixing two different concepts:

### Action-specific audit snapshots
- `returned_to_preparation_*`
- `chef_decision_*`

### Current reopen explanation
- which action is currently responsible for the dossier being in `AWAITING_COMPLEMENT`

The page was trying to answer:

> “Why is this dossier currently reopened?”

But the model only gave it multiple competing snapshot families.

---

# 11) Final Clean Solution: Canonical Awaiting-Complement Context

The final Day 32 refinement introduced a clean separation between:

- **action-specific audit fields**
- **current reopen context**

## 11.1 Second migration added

A second migration was introduced:

- `2026_04_15_014521_add_awaiting_complement_context_to_dossiers_table.php`

New fields added on `dossiers`:

- `awaiting_complement_source`
- `awaiting_complement_by`
- `awaiting_complement_at`
- `awaiting_complement_note`

## 11.2 Why this is the clean solution

These fields answer only one question:

> **Why is the dossier currently in `AWAITING_COMPLEMENT`?**

### Allowed sources
- `CLAIMS_MANAGER_RETURN`
- `SUPERVISOR_COMPLEMENT_REQUEST`

This creates a single canonical current-state explanation, while preserving older fields for their own audit meaning:

- `returned_to_preparation_*` = Claims Manager return audit
- `chef_decision_*` = supervisor decision audit

This is much cleaner than clearing old fields as the main fix or making the frontend guess the latest note by timestamp.

## 11.3 `Dossier` model updated again

`Dossier.php` was extended with:

- new fillable `awaiting_complement_*` fields
- `awaiting_complement_at` cast
- new relation:
  - `awaitingComplementBy()`

## 11.4 `DossierController` updated again

The controller was updated so:

- `returnToPreparation()` now writes:
  - `awaiting_complement_source = 'CLAIMS_MANAGER_RETURN'`
  - actor
  - timestamp
  - note

- `submit()` clears the current reopen context once the dossier leaves the reopened-preparation state

- dossier summary/detail payloads now expose:
  - `awaiting_complement_source`
  - `awaiting_complement_by`
  - `awaiting_complement_at`
  - `awaiting_complement_note`
  - `awaiting_complement_user`

## 11.5 `DossierEscalationController` updated too

The escalation controller was updated so:

- `requestComplement()` now writes:
  - `awaiting_complement_source = 'SUPERVISOR_COMPLEMENT_REQUEST'`
  - actor
  - timestamp
  - note

- escalation actions that leave the reopen state clear the canonical reopen context when appropriate

This made the reopened-preparation explanation consistent across both entry paths.
