# 3) Transactional Integrity & File I/O Refactoring

## 3.1 Problem Identified

File storage previously occurred inside a database transaction.

Risks:

* Long-held DB connections.
* Scalability bottlenecks.
* Orphaned files if the transaction fails.
* Mixed I/O and persistence responsibilities.

---

## 3.2 Refactor Applied

Separated the I/O operation from the database transaction to ensure clean rollback:

```php
$path = Storage::putFileAs(...);

try {
    DB::beginTransaction();

    $document = Document::create([...]);

    ProcessDocumentJob::dispatch($document->id)->afterCommit();

    DB::commit();
} catch (\Exception $e) {
    DB::rollBack();
    Storage::delete($path);
    throw $e;
}
```

---

## 3.3 Improvements Achieved

* File written before transaction begins.
* Transaction window minimized.
* Clean separation of concerns (I/O vs persistence).
* Guaranteed cleanup on failure.

# 4) AIService Resource Leak Prevention

## 4.1 Problem Identified

The `fopen()` handle used to send files to the AI microservice was not guaranteed to close if the HTTP attachment failed.

Risk: file descriptor exhaustion and memory leaks.

---

## 4.2 Fix Implemented

Wrapped the FastAPI HTTP call inside a `try/finally` block:

```php
$handle = fopen($fullPath, 'r');

try {
    $response = Http::attach('file', $handle, basename($filePath))
        ->post(...);
} finally {
    fclose($handle);
}
```

---

## 4.3 Result

* Guaranteed file handle closure.
* Strict resource lifecycle control.
* Memory safety enforced.

# 8) Frontend Stabilization – Deterministic Date Normalization

## 8.1 Problem Identified

The `DocumentValidation` component previously relied on JavaScript’s `new Date()` constructor to normalize OCR-extracted dates.

This introduced:

* Browser-dependent parsing.
* Incorrect handling of `DD-MM-YYYY`.
* Potential timezone shifts with `toISOString()`.
* Non-deterministic OCR behavior.

For a document processing system, deterministic formatting is mandatory.

---

## 8.2 Fix Implemented

Replaced dynamic parsing with strict manual normalization:

```javascript
const normalizeDate = (value) => {
  if (!value) return null;

  const raw = String(value).trim();

  if (/^\d{4}-\d{2}-\d{2}$/.test(raw)) {
    return raw;
  }

  const clean = raw.replace(/\//g, '-');
  const parts = clean.split('-');

  if (parts.length === 3) {
    const [a, b, c] = parts;

    if (a.length === 2 && c.length === 4) {
      return `${c}-${b}-${a}`;
    }

    if (a.length === 4) {
      return `${a}-${b}-${c}`;
    }
  }

  return null;
};
```

---

## 8.3 Result

* Deterministic frontend behavior.
* Support for:

  * `YYYY-MM-DD`
  * `DD-MM-YYYY`
  * `DD/MM/YYYY`
* No timezone inconsistencies.
* OCR-safe validation layer.
