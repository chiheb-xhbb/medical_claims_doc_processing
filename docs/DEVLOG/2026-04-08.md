# 1) Objective of Day 29

After Day 28, the hierarchical review workflow was already complete end-to-end:

- backend role/state support existed
- frontend role-aware integration existed
- the bell worked
- returned dossier re-review worked
- complement reopening worked
- payload contracts were aligned

However, one structural issue remained in the codebase:

- naming was still mixed between French business labels, English UI text, and partially normalized enum values

This created several architectural weaknesses:

- duplicated vocabulary across backend and frontend
- reduced readability for the jury
- higher refactor risk for future i18n support
- potential RBAC/workflow drift between Laravel and React
- leftover legacy enum references capable of causing runtime failures after enum renaming

Day 29 was therefore dedicated to a **controlled naming refactor sprint**, not to a new feature.

The goal was to:

- freeze one canonical English naming layer for internal code values
- preserve the validated business model (`Dossier`, `Rubrique`) without risky aggregate renames
- align backend enums, controllers, requests, migration data, and frontend constants
- improve visible UI wording to polished English
- keep backend-compatible payload keys and endpoint paths where required
- stabilize the application after the rename by removing remaining legacy enum references

# 2) Core Naming Decision Locked on Day 29

A key architectural distinction was formalized:

## 2.1 Internal code values must be stable and English

Internal values now use canonical English naming for:

- roles
- dossier workflow states
- frontend constants
- route protection logic
- UI action gating

This creates a stable technical vocabulary across the stack.

## 2.2 UI labels are presentation-layer values

The visible UI can later be translated independently of enum values.

This is important for future i18n support:

- internal value stays stable
- visible label can change by language

Example:

- internal role value: `SUPERVISOR`
- English label: `Supervisor`
- future French label: `Chef hiérarchique`

This avoids coupling persistence and workflow logic to display language.

## 2.3 Domain nouns kept unchanged internally

An important Day 29 decision was also preserved:

- `Dossier` remains `Dossier` in backend code/model naming
- `Rubrique` remains `Rubrique` in backend code/model naming

These are already established aggregate/business concepts in the application.

Renaming them at this stage would create disproportionate risk in:

- models
- table names
- relations
- API contracts
- existing documentation

So Day 29 focused on the inconsistent naming layer, not the aggregate roots.

# 3) Final Canonical Vocabulary Introduced

## 3.1 User roles

The old mixed vocabulary was normalized to:

- `AGENT`
- `CLAIMS_MANAGER`
- `SUPERVISOR`
- `ADMIN`

### Role mapping applied

- `GESTIONNAIRE` → `CLAIMS_MANAGER`
- `CHEF_HIERARCHIQUE` → `SUPERVISOR`

## 3.2 Dossier statuses

The dossier workflow naming was normalized to:

- `RECEIVED`
- `IN_PROGRESS`
- `UNDER_REVIEW`
- `IN_ESCALATION`
- `AWAITING_COMPLEMENT`
- `PROCESSED`

### Status mapping applied

- `TO_VALIDATE` → `UNDER_REVIEW`
- `EN_DEROGATION` → `IN_ESCALATION`
- `COMPLEMENT_ATTENDU` → `AWAITING_COMPLEMENT`

## 3.3 Business semantics preserved

Even after renaming, the actual business workflow did **not** change.

The state machine remains:

```text
RECEIVED
   ↓
IN_PROGRESS
   ↓
UNDER_REVIEW
   ├── normal review success → PROCESSED
   └── exceptional case → IN_ESCALATION
```

Then:

```text
IN_ESCALATION
   ├── approve escalation → PROCESSED
   ├── return to Claims Manager → UNDER_REVIEW
   └── request complement → AWAITING_COMPLEMENT
```

And finally:

```text
AWAITING_COMPLEMENT
   ↓
preparation owner completes missing elements
   ↓
UNDER_REVIEW
```

So Day 29 changes the vocabulary layer, **not** the validated workflow behavior.

# 6) Database – Existing Data Normalization

A dedicated migration was created:

- `2026_04_07_224537_normalize_roles_and_dossier_status_values.php`

## 6.1 Why this migration was mandatory

Changing enums in code is not enough if the database still stores old values.

Without a data migration, the system can fail at runtime because Laravel enum casting expects the new canonical values while old rows still contain legacy strings.

## 6.2 Data updates performed

The migration normalized persisted values in:

- `dossiers.status`
- `users.role`

### Applied mappings

For dossiers:

- `TO_VALIDATE` → `UNDER_REVIEW`
- `EN_DEROGATION` → `IN_ESCALATION`
- `COMPLEMENT_ATTENDU` → `AWAITING_COMPLEMENT`
- legacy `RECU` → `RECEIVED`

For users:

- `GESTIONNAIRE` → `CLAIMS_MANAGER`
- `CHEF_HIERARCHIQUE` → `SUPERVISOR`

## 6.3 Result

The codebase and persisted data are now aligned on the same naming layer.

This removes the most dangerous inconsistency class of this refactor:

- code expecting new enum values
- database still containing old values

# 11) Final Runtime Regression and Resolution

Day 29 uncovered one important regression after the enum rename.

## 11.1 Problem encountered

After backend enum updates, some API calls began failing with HTTP 500.

The concrete Laravel runtime error was:

```text
Undefined constant App\Enums\UserRole::GESTIONNAIRE
```

## 11.2 Root cause

The enum files had been renamed, but some backend files still contained old enum constant references such as:

- `UserRole::GESTIONNAIRE`
- `UserRole::CHEF_HIERARCHIQUE`
- `DossierStatus::TO_VALIDATE`

This created a partial-refactor state where:

- enums had changed
- some controllers/requests/models were still calling removed constants

## 11.3 Resolution applied

The remaining legacy enum references were removed from the affected backend files, including document-related authorization/review code and residual domain/workflow logic.

After the correction:

- Laravel caches/autoload were cleared
- the application was re-tested
- document and dossier routes returned to normal behavior

## 11.4 Architectural lesson

This confirmed an important engineering point for the jury:

> enum renaming is not a cosmetic change; it is a contract change across authorization, workflow state checks, persistence, and runtime casting.

So the stabilization step was a necessary part of the refactor, not an accidental detour.
