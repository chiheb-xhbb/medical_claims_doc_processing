## 2) Workflow Normalization – Removal of `EXPORTED`

### 2.1 Problem identified

The dossier lifecycle still contained the `EXPORTED` state even though it no longer brought real métier value in the current reimbursement workflow.

Keeping it created unnecessary complexity in:

- business reasoning
- frozen-state logic
- visibility rules
- controller conditions

### 2.2 Decision taken

The lifecycle was simplified to keep only the useful states:

- `RECEIVED`
- `IN_PROGRESS`
- `TO_VALIDATE`
- `PROCESSED`

### 2.3 Result

This made the dossier workflow easier to understand and more aligned with the current métier scope:

```text
RECEIVED → IN_PROGRESS → TO_VALIDATE → PROCESSED
```

It also reduced unnecessary branching in the backend and clarified that `PROCESSED` is the real terminal business state for the current project scope.

---

## 3) Traceability Refactor – From Generic Audit to Action-Based Audit

### 3.1 Problem identified

A generic idea such as `verified_by` was not strong enough architecturally because it does not indicate **which action** was actually performed.

For a workflow-oriented system, the audit should answer concrete questions such as:

- who submitted the dossier?
- who processed the dossier?
- who rejected the rubrique?
- when did each of these actions happen?

### 3.2 Design decision

Instead of introducing vague fields, the traceability model was made action-based.

For `dossiers`, the following fields were added:

- `submitted_by`
- `processed_by`
- `processed_at`

For `rubriques`, the following fields were added:

- `rejected_by`
- `rejected_at`

### 3.3 Migration layer

A dedicated migration was created to extend existing tables without breaking the current structure. This keeps the project aligned with a progressive schema evolution approach instead of rewriting old persistence decisions.

### 3.4 Controller integration

The dossier workflow actions were updated so that:

- dossier submission now stores `submitted_by` and `submitted_at`
- dossier processing now stores `processed_by` and `processed_at`
- rubrique whole rejection now stores `rejected_by` and `rejected_at`

### 3.5 Model integration

The model layer was also completed so the new fields are first-class citizens in the domain:

- datetime casts added where necessary
- relations for submitter / processor / rejector added
- rubrique rejection traceability moved into the correct business method rather than being left implicit

### 3.6 Result

The system now uses a much more defensible audit style:

- dossier submitted by X at Y
- dossier processed by X at Y
- rubrique rejected by X at Y

This is significantly more professional than scattering a generic `verified_by` field everywhere.

## 5) Documents Listing – Backend Filters and Sorting

### 5.1 Objective

The documents list already existed, but its consultation behavior needed to become more product-like.

A professional list page should not rely on frontend-only filtering of the current page. It should expose real query capabilities at API level.

### 5.2 Filters added

`DocumentController@index` was expanded to support:

- `search`
- `status`
- `from_date`
- `to_date`
- `per_page`

The search behavior supports:

- filename-based search
- direct numeric ID search

### 5.3 Sorting added

Sorting support was added with:

- `sort_by`
- `sort_direction`

Allowed sort fields were intentionally restricted to:

- `id`
- `status`
- `created_at`

A strict allow-list was also applied on direction:

- `asc`
- `desc`

### 5.4 Result

The documents endpoint is now suitable for a real filtered and sortable data table, while keeping backend control over query safety and consistency.

## 9) Frontend Stability – Removal of the “Full Page Refresh” Feeling

### 9.1 Problem identified

Even though the application is an SPA, list actions still felt visually heavy because loading and error states could replace the whole page structure.

This created a poor UX sensation similar to a complete page refresh.

### 9.2 Fix implemented

The pages were refactored so that:

- page header remains mounted
- filter bar remains mounted
- only the table section updates
- empty state remains inside the table container
- loading is shown through a local table overlay and skeleton rows instead of replacing the full page

### 9.3 Result

Applying filters, sorting, and switching pages now feels much smoother and more professional.
