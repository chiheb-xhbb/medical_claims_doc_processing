# 1) Objective of Day 36

After Day 35, the application already had:

- stabilized dossier workflow semantics
- improved branding and navbar polish
- shared pagination refinement
- semantic toast feedback for local page actions

However, the system still lacked a **real persistent in-app notification layer**.
Important workflow events were visible only through immediate page feedback or through manual user refresh, which meant there was no durable cross-role notification trace for:

- dossier submission to review
- dossier resubmission after complement
- escalation to supervisor
- return from supervisor to claims manager
- return to preparation
- complement request to the preparation owner

Day 36 therefore introduced a dedicated notification subsystem with clear design goals:

- **persistent** (database-backed)
- **workflow-triggered** (Laravel remains the source of truth)
- **recipient-based** (notifications target the correct business actor)
- **safe** (notification failure must never break the workflow)
- **SPA-friendly** (relative `action_url` values)
- **lightweight** (polling instead of WebSockets)

This was intentionally scoped as a production-like internal feature, not as a generic public notification platform.

# 2) Architecture and Scope Decisions

The notification system was designed around a few locked decisions that keep the implementation coherent with the rest of the project.

## 2.1 Laravel remains the workflow authority

Notifications are created by Laravel **only after a successful workflow transition**.
The frontend never decides who should be notified and never constructs business notifications by itself.

That keeps the architecture aligned with the project rule already established earlier:

- React displays state
- Laravel owns business workflow and access rules

## 2.2 Polling instead of WebSockets

No Echo, Pusher, Reverb, or WebSockets were introduced.
Instead, the bell follows a small, controlled polling strategy:

- unread count refresh every **30 seconds**
- recent notifications fetched when the bell opens
- recent list refresh every **60 seconds** while the panel stays open

This is the right engineering tradeoff for a late-stage PFE: lower risk, easier validation, and no deployment complexity explosion.

## 2.3 Naming collision avoidance

The model is named:

```php
App\Models\AppNotification
```

and **not** `Notification`.

This avoids conflict with Laravel’s native notification classes and keeps the domain model explicit.

## 2.4 Relative SPA navigation

Each notification stores a relative path such as:

```php
'action_url' => '/dossiers/' . $dossier->id,
```

This keeps navigation smooth on the React side and avoids hardcoding absolute environment URLs.

# 3) Database Foundation – `app_notifications`

A new table named `app_notifications` was added as the persistent storage layer.
It captures both the recipient and the workflow context of the event.

Representative migration excerpt:

```php
Schema::create('app_notifications', function (Blueprint $table) {
    $table->id();

    $table->foreignId('user_id')->constrained('users')->cascadeOnDelete();
    $table->foreignId('actor_id')->nullable()->constrained('users')->nullOnDelete();

    $table->foreignId('dossier_id')->nullable()->constrained('dossiers')->nullOnDelete();
    $table->foreignId('rubrique_id')->nullable()->constrained('rubriques')->nullOnDelete();
    $table->foreignId('document_id')->nullable()->constrained('documents')->nullOnDelete();

    $table->string('type', 100);
    $table->string('title', 255);
    $table->text('message');
    $table->string('action_url')->nullable();

    $table->boolean('is_read')->default(false);
    $table->timestamp('read_at')->nullable();
    $table->json('meta')->nullable();

    $table->timestamps();

    $table->index(['user_id', 'is_read', 'created_at'], 'app_notifications_user_read_created_idx');
    $table->index('dossier_id');
    $table->index('type');
});
```

### Why this schema matters

This structure makes the notification rows useful for both UI and traceability:

- `user_id` identifies the recipient
- `actor_id` identifies who triggered the workflow action
- `dossier_id` links the row to the business entity
- `type` makes the event machine-readable
- `title` and `message` provide user-facing meaning
- `action_url` makes the bell actionable
- `meta` preserves lightweight structured context
- `is_read` / `read_at` support the notification center behavior

The final result is a table that is simple enough for the PFE scope but still strong enough to defend architecturally.

# 4) Backend Model and Notification Service

## 4.1 `AppNotification` model

A dedicated model was introduced with:

- proper `$fillable` fields
- casts for `is_read`, `read_at`, and `meta`
- relations to `user`, `actor`, and `dossier`

This makes the notification resource easy to query, serialize, and extend later.

## 4.2 Centralized `NotificationService`

Instead of duplicating notification creation logic across controllers, the work was centralized in `NotificationService`.

Its responsibilities are:

- normalize notification payloads
- create one row per recipient
- resolve recipient groups by role
- preserve a stable `meta` shape
- log failures without breaking the main business action

Representative service excerpt:

```php
public function notifyUsers(array $userIds, array $payload): void
{
    $uniqueUserIds = collect($userIds)
        ->filter(fn ($id) => ! is_null($id))
        ->map(fn ($id) => (int) $id)
        ->unique()
        ->values();

    if ($uniqueUserIds->isEmpty()) {
        Log::warning('NotificationService: no recipients found.', [
            'type' => $payload['type'] ?? null,
            'dossier_id' => $payload['dossier_id'] ?? null,
        ]);
        return;
    }

    $rows = $uniqueUserIds->map(function (int $userId) use ($payload) {
        return [
            'user_id' => $userId,
            'actor_id' => $payload['actor_id'] ?? null,
            'dossier_id' => $payload['dossier_id'] ?? null,
            'type' => $payload['type'],
            'title' => $payload['title'],
            'message' => $payload['message'],
            'action_url' => $payload['action_url'] ?? null,
            'is_read' => false,
            'read_at' => null,
            'meta' => isset($payload['meta']) ? json_encode($payload['meta']) : null,
            'created_at' => now(),
            'updated_at' => now(),
        ];
    })->all();

    try {
        AppNotification::insert($rows);
    } catch (\Throwable $e) {
        Log::error('NotificationService: failed to create notifications.', [
            'error' => $e->getMessage(),
            'type' => $payload['type'] ?? null,
            'dossier_id' => $payload['dossier_id'] ?? null,
        ]);
    }
}
```

### Important engineering decision

Notification creation is **best-effort**:

- empty recipient groups log a warning
- insert failure logs an error
- the workflow transition itself still succeeds

This preserves the correct priority:

1. workflow integrity first
2. notification delivery second

## 4.3 Role-based recipient helpers

The service provides dedicated helper methods:

- `notifyPreparationOwner(...)`
- `notifyAllClaimsManagers(...)`
- `notifySupervisors(...)`

This keeps controllers focused on business transitions rather than on recipient selection mechanics.

# 5) Notification API Layer

A dedicated `NotificationController` was introduced to expose bell data safely to React.

Implemented endpoints:

- `GET /api/notifications`
- `GET /api/notifications/unread-count`
- `POST /api/notifications/{notification}/read`
- `POST /api/notifications/read-all`

These endpoints are registered inside the existing protected API group:

```php
Route::prefix('notifications')->group(function () {
    Route::get('/', [NotificationController::class, 'index']);
    Route::get('/unread-count', [NotificationController::class, 'unreadCount']);
    Route::post('/read-all', [NotificationController::class, 'markAllRead']);
    Route::post('/{notification}/read', [NotificationController::class, 'markRead']);
});
```

### Controller behavior

The controller keeps the API intentionally narrow:

- `index()` returns the current user’s notifications, latest first
- `unreadCount()` supports lightweight badge refresh
- `markRead()` enforces ownership before updating one row
- `markAllRead()` updates unread rows for the authenticated user only

Representative access-control guard:

```php
if ((int) $notification->user_id !== (int) $user->id) {
    return response()->json([
        'message' => 'You are not allowed to access this notification.',
    ], 403);
}
```

This keeps the bell secure and avoids cross-user data leakage.

# 6) Workflow Wiring – Notification Emission Points

The most important part of Day 36 was not the bell UI itself.  
It was the correct wiring of **workflow events to recipients**.

## 6.1 `DossierController`

Two key transitions were wired here:

### First submission vs resubmission

When a dossier moves to `UNDER_REVIEW`, the system now distinguishes:

- first submission → `DOSSIER_SUBMITTED_FOR_REVIEW`
- reopened submission after complement → `DOSSIER_RESUBMITTED_FOR_REVIEW`

Representative excerpt:

```php
$wasResubmission = $dossier->status === DossierStatus::AWAITING_COMPLEMENT;

DB::afterCommit(function () use ($user, $dossier, $wasResubmission) {
    $this->notificationService->notifyAllClaimsManagers(
        actor: $user,
        dossier: $dossier,
        type: $wasResubmission
            ? 'DOSSIER_RESUBMITTED_FOR_REVIEW'
            : 'DOSSIER_SUBMITTED_FOR_REVIEW',
        title: $wasResubmission
            ? 'Case file resubmitted for review'
            : 'Case file submitted for review',
        message: $wasResubmission
            ? "Case file {$dossier->numero_dossier} has been resubmitted for review."
            : "Case file {$dossier->numero_dossier} has been submitted for review."
    );
});
```

### Return to preparation

When a claims manager sends a dossier back to preparation, the preparation owner now receives:

- `DOSSIER_RETURNED_TO_PREPARATION`

This closes the feedback loop for the dossier creator.

## 6.2 `DossierEscalationController`

Three supervisor-side actions were wired after commit.

### Escalation to supervisor

```php
DB::afterCommit(function () use ($user, $fresh) {
    $this->notificationService->notifySupervisors(
        actor: $user,
        dossier: $fresh,
        type: 'DOSSIER_ESCALATED',
        title: 'Case file escalated for review',
        message: "Case file {$fresh->numero_dossier} has been escalated and requires your review."
    );
});
```

### Return to claims manager

A supervisor can now send the dossier back for additional review, creating:

- `DOSSIER_RETURNED_TO_CLAIMS_MANAGER`

### Complement request to preparation owner

A supervisor can now request complement directly, creating:

- `DOSSIER_COMPLEMENT_REQUESTED`

This was an essential part of the day because these transitions already existed in the business workflow, but the application had no persistent cross-role notification trace for them before.

# 7) Frontend Service Layer

The frontend integration starts with a dedicated service module:

```js
import api from './api';

export async function getNotifications(page = 1, perPage = 10) {
  const response = await api.get('/notifications', {
    params: { page, per_page: perPage },
  });
  return response.data;
}

export async function getUnreadNotificationCount() {
  const response = await api.get('/notifications/unread-count');
  return response.data;
}

export async function markNotificationRead(notificationId) {
  const response = await api.post(`/notifications/${notificationId}/read`);
  return response.data;
}

export async function markAllNotificationsRead() {
  const response = await api.post('/notifications/read-all');
  return response.data;
}
```

### Why this matters

The bell does not call raw endpoints inline everywhere.
This keeps network access:

- centralized
- easier to maintain
- consistent with the rest of the frontend architecture

# 8) Bell Component and Navbar Integration

## 8.1 Role-aware bell visibility

The new `NotificationBellButton` is visible only for the intended business actors:

```jsx
const BELL_VISIBLE_ROLES = new Set([
  USER_ROLES.AGENT,
  USER_ROLES.CLAIMS_MANAGER,
  USER_ROLES.SUPERVISOR,
]);
```

`ADMIN` intentionally has no bell in this scope.

## 8.2 Polling and bell behavior

The component handles:

- unread count polling every 30 seconds
- recent notification list loading on open
- 60-second list refresh while open
- mark-one-read before navigation
- mark-all-read from the panel
- relative time rendering for better readability

Representative polling excerpt:

```jsx
useEffect(() => {
  if (!canShowBell) return;

  loadUnreadCount();
  const id = window.setInterval(loadUnreadCount, 30_000);
  return () => window.clearInterval(id);
}, [canShowBell, loadUnreadCount]);
```

Representative on-open behavior:

```jsx
const handleToggleBell = async () => {
  const opening = !isOpen;
  setIsOpen(opening);

  if (opening) {
    await Promise.all([loadUnreadCount(), loadRecentNotifications()]);
  }
};
```

## 8.3 Click-to-read and navigation

When the user clicks a notification:

- unread notifications are marked as read first
- UI state is updated immediately
- the dropdown closes
- navigation goes to the dossier detail page

Representative excerpt:

```jsx
if (isUnread) {
  const res = await markNotificationRead(notification.id);
  const updated = res?.notification;

  setNotifications((prev) =>
    prev.map((n) =>
      n.id === notification.id ? { ...n, ...(updated || {}), is_read: true } : n
    )
  );
  setUnreadCount((prev) => Math.max(0, prev - 1));
}

setIsOpen(false);
navigate(url);
```

## 8.4 Navbar integration

The old specialized `SupervisorBellButton` was removed.
The new generic bell is now integrated directly into the stabilized navbar shell, before the user profile dropdown.

This keeps the user area coherent and prevents the notification center from feeling like a one-role exception.
