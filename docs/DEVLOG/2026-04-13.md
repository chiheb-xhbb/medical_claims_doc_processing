- `backend/app/Http/Controllers/Api/AuthController.php`

### Behavior implemented

The authenticated user can now:
- submit current password
- submit new password
- confirm new password

Validation rules enforce:
- current password required
- new password required
- minimum length
- confirmation match
- different from current password

### Endpoint added

Inside the authenticated route group in `backend/routes/api.php`:


### Why this belongs in `AuthController`

This controller already owned:
- `login`
- `logout`
- `me`

So self password change belongs naturally to the same responsibility area.

### Result

The application now supports a serious first level of user-controlled account security without changing the existing authentication architecture.

## 5) Frontend – Self Password Change Integration

## 5.1 Auth service extended

`frontend/src/services/auth.js` was updated with a dedicated helper:

```js
export async function changePassword(currentPassword, newPassword, newPasswordConfirmation) {
  const response = await api.post('/auth/change-password', {
    current_password: currentPassword,
    password: newPassword,
    password_confirmation: newPasswordConfirmation,
  });

  return response.data;
}
```

### Why this matters

This keeps password-change logic out of UI components and preserves the same service-layer pattern already used for login/logout/current user flows.

## 6) Backend – Admin Password Reset

## 6.1 `AdminUserController` extended with `resetPassword()`

A new admin-only account-management action was added to:

- `backend/app/Http/Controllers/Api/AdminUserController.php`

### Endpoint added

Inside the authenticated admin route group:

```php
Route::patch('/users/{user}/password', [AdminUserController::class, 'resetPassword']);
```

### Behavior implemented

An admin can now reset another user's password by submitting:
- `password`
- `password_confirmation`

The method:
- enforces admin access through the controller’s existing admin-check pattern
- validates the new password
- wraps the change in a database transaction
- locks the target user row with `lockForUpdate()`
- hashes the password securely
- revokes the target user’s existing Sanctum tokens

### Why token revocation was important

Without token revocation, an already-connected user could remain authenticated even after an admin reset.

That would weaken the security meaning of the reset operation.

By deleting the user’s active tokens after the reset, the implementation becomes aligned with the stricter account-control posture already used elsewhere in the controller.

### Result

The project now has a serious admin password reset story, not just a cosmetic frontend action.

## 8) Modal Infrastructure Stabilization

One of the most important technical corrections of Day 31 happened in:

- `frontend/src/components/WorkspaceModalShell.jsx`

## 8.1 Focus-loss bug discovered

During implementation, a severe UX issue appeared:

> typing one character in modal inputs caused focus to leave the field

This made the form practically unusable.

## 8.2 Root cause

The modal shell used an effect for keyboard handling and focus trapping whose dependency array included unstable values such as `onClose` and `isBusy`.

Because parent components passed fresh inline callbacks during re-renders, the effect could tear down and restore focus unexpectedly while the user was typing.

## 8.3 Fix applied

The modal shell was stabilized using refs to hold the latest values while keeping the main focus-trap effect tied only to actual modal-open state changes.

### Result

Typing in modal inputs is now stable and normal.

This was not cosmetic polish; it was a critical usability defect that had to be fixed before the account-management flow could be considered complete.
