## 1) Objective of Day 26

After Day 25, the AI layer was already stabilized and the application had a mature backend/frontend foundation.

However, two important needs remained:

- a clarified business rule from the supervisor: **Gestionnaire can also upload documents and create dossiers**
- a need to make the browser workflow cleaner, faster, and more professional by reducing unnecessary top-level pages and polishing the interface

Day 26 was therefore treated as a **consolidation and UX refactor day**, not as a random cosmetic pass.

The work focused on four parallel goals:

- align backend ownership rules with the clarified role behavior
- align the frontend with that backend behavior
- strengthen the visual quality of the main workspaces
- move lightweight creation actions into modal-first flows while preserving the existing architecture and business logic

---

## 2) Backend – Gestionnaire Ownership Coherence

### 2.1 Problem clarified

The initial RBAC direction already separated the roles:

- `AGENT`
- `GESTIONNAIRE`
- `ADMIN`

But the supervisor clarified an important practical rule:

- a **Gestionnaire** must also be able to upload documents
- a **Gestionnaire** must also be able to create dossiers

That meant the backend could no longer treat dossier/document ownership actions as if they belonged only to `AGENT`.

### 2.2 Dossier ownership logic updated

`DossierController.php` was adjusted so the ownership helper now treats **Gestionnaire** like **Agent** for **own dossier** management.

In practical terms, ownership remains based on `created_by`, but the allowed roles now include:

- `AGENT`
- `GESTIONNAIRE`
- `ADMIN`

### 2.3 Safe document deletion logic updated

`DocumentController.php` was also updated so **Gestionnaire** can delete **own eligible documents** under the same safety rules already used for the other roles.

The logic still preserves the existing constraints:

- deletion remains limited to the allowed technical statuses
- deletion remains blocked when the document is already attached
- ownership still depends on `user_id`
- `ADMIN` still keeps global supervision behavior

### 2.4 Result

This corrected a real functional incoherence:

- a user who can create and upload as Gestionnaire must also be treated coherently for ownership-based cleanup and management of his own resources

## 3) Frontend Alignment with the New Backend Rules

### 3.3 Auth state synchronization polish

Role/user-dependent UI state was also stabilized by aligning the list pages with the same `AUTH_CHANGED_EVENT` synchronization pattern, reducing stale role-display behavior after auth changes.

### 3.4 Result

The browser behavior became more coherent with the backend:

- Gestionnaire can now see and use the actions that match the clarified business rule
- stale lifecycle logic was removed
- role-driven list behavior became cleaner and more stable

## 5) Workflow Refactor – Upload and Create Dossier Become Modal-First

### 5.1 Product decision

Two actions were identified as **workspace actions**, not true standalone top-level destinations:

- **Upload Documents**
- **Create Dossier**

Instead of keeping them as separate primary pages in the browser workflow, Day 26 refactored them into modal-first experiences launched directly from their respective workspaces.

### 5.2 Upload Documents refactor

The Upload workflow was redesigned so that:

- the top-level **Upload** navbar section is removed
- the Documents page becomes the primary document workspace
- clicking **Upload Documents** now opens a large premium modal
- the upload logic is reused instead of duplicated

To keep the code clean, the upload UI was extracted into:

- `DocumentUploadForm.jsx`
- `DocumentUploadModal.jsx`

The old route is not hard-broken; it now safely redirects:

- `/documents/upload` → `/documents`

### 5.3 Create Dossier refactor

The same logic was applied to dossier creation:

- Dossiers remains the main workspace
- clicking **New Dossier** opens a modal instead of a full page
- the form logic is extracted cleanly into reusable components

This resulted in:

- `DossierCreateForm.jsx`
- `DossierCreateModal.jsx`

The old route is now safely redirected:

- `/dossiers/create` → `/dossiers`

### 5.4 Shared modal infrastructure

To avoid two ad-hoc modal implementations, a shared shell was introduced:

- `WorkspaceModalShell.jsx`
- `workspace-modals.css`

This shared shell centralizes:

- backdrop
- width variants
- header structure
- close action
- focus trapping
- escape handling
- busy-state protection
- common premium modal language

### 5.5 Business-flow consistency

The two modal flows were kept consistent with their actual business role:

#### Upload Documents
- open modal from Documents workspace
- upload within the modal
- close on success
- refresh the documents list inline
- stay inside the Documents workspace

#### Create Dossier
- open modal from Dossiers workspace
- create dossier inside the modal
- close on success
- navigate directly to the newly created dossier detail page

This was the cleanest workflow choice because dossier creation is usually the entry point toward preparation.

## 6) Upload Modal Visual Refinement

After the first modal refactor, the dossier modal already felt balanced, but the upload modal still looked too stretched horizontally.

### 6.1 Root cause

The upload shell and its internal content occupied too much horizontal space:

- the dialog width was too generous for the real amount of content
- the drop zone and full-width controls formed a large flat slab
- the visual rhythm was weaker than in the dossier modal

### 6.2 Refinement applied

The upload modal was refined through upload-specific shared modal rules:

- a narrower upload modal width
- centered upload content column
- calmer internal spacing
- better vertical rhythm in the drop zone
- better balance between file actions, upload button, and help text

### 6.3 Result

The upload modal now feels:

- less stretched
- more premium
- more coherent with the dossier modal
- better integrated into the overall design language
