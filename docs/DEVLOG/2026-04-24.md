# 2) Architecture and Scope Decisions

## 2.1 Laravel remains the source of financial truth

A key decision was to avoid calculating reimbursement totals in React.

The financial summary is derived from existing persisted data:

- dossiers;
- rubriques;
- documents;
- extractions;
- document decision statuses.

Therefore, no new columns were needed. The implementation uses computed backend response fields instead of a schema change.

This preserves the project’s architecture rule:

> Laravel owns workflow, persistence, RBAC, business totals, and API truth. React only renders the response.

## 2.2 No migrations for derived totals

The following values are derived from existing data:

- `requested_total`;
- `accepted_total`;
- `rejected_total`;
- `final_reimbursable_total`.

Because they can be computed from document decisions and extraction totals, adding database columns would have created synchronization risk.

For example, storing `accepted_total` and `rejected_total` would require updating aggregates after every decision change, return cycle, complement resubmission, or final process action. The safer design is to compute the values from the current backend state and return them in the dossier payload.

## 2.3 Current context is not workflow history

Another important decision was to keep the following concepts separate:

- **Active Workflow Context**: explains the current actionable situation;
- **Workflow History**: shows the immutable chronological audit trail.

This prevents the page from becoming noisy and avoids duplicating the same return/complement/escalation information in several cards.

## 2.4 Frontend polish stayed frontend-only

Most Day 39 tasks were presentation improvements. These did not require touching:

- Laravel workflow transitions;
- role policies;
- database schema;
- FastAPI OCR/extraction service;
- document-processing jobs.

The scope was kept deliberately narrow to avoid late-stage regressions.

# 3) Business Financial Summary

## 3.1 Problem

The dossier summary card previously exposed technical/internal total concepts:

- `Requested Total`;
- `Current Total`;
- `Display Total`;
- `Stored Total`.

For a reimbursement workflow, `Display Total` and `Stored Total` are not business-friendly labels. They sound like debug or implementation concepts.

The UI needed to show business values instead:

- requested amount;
- accepted amount;
- rejected amount;
- final reimbursable amount.

## 3.2 Backend financial summary payload

The Laravel dossier response was extended with a computed financial summary structure:

```json
"financial_summary": {
  "requested_total": 121.000,
  "accepted_total": 67.000,
  "rejected_total": 54.000,
  "final_reimbursable_total": null
}
```

For processed dossiers, the final amount becomes available:

```json
"financial_summary": {
  "requested_total": 121.000,
  "accepted_total": 67.000,
  "rejected_total": 54.000,
  "final_reimbursable_total": 67.000
}
```

The legacy fields were preserved for compatibility where needed, but the visible UI now relies on the business-facing financial summary.

## 3.3 Dossier model helpers

The financial logic was centralized in the `Dossier` model instead of being duplicated across controllers.

The final responsibility is:

- `requested_total`: sum of validated attached documents;
- `accepted_total`: sum of validated attached documents with decision `ACCEPTED`;
- `rejected_total`: sum of validated attached documents with decision `REJECTED`;
- `final_reimbursable_total`: `null` until the dossier is `PROCESSED`; once processed, uses the frozen `montant_total`.

This keeps the business interpretation close to the dossier aggregate.

Representative structure:

```php
public function getFinancialSummary(): array
{
    return [
        'requested_total' => $this->getRequestedTotal(),
        'accepted_total' => $this->getAcceptedTotal(),
        'rejected_total' => $this->getRejectedTotal(),
        'final_reimbursable_total' => $this->getFinalReimbursableTotal(),
    ];
}

public function toFinancialTotalsPayload(): array
{
    $financialSummary = $this->getFinancialSummary();

    return [
        'requested_total' => $financialSummary['requested_total'],
        'current_total' => $financialSummary['accepted_total'],
        'display_total' => $this->getDisplayTotal(),
        'financial_summary' => $financialSummary,
    ];
}
```

The important point is that the UI receives clean business totals while compatibility fields remain available.

## 3.4 Document extraction helper

Because totals are derived from the latest extraction, `Document` was given a clean helper for resolving the latest extracted `total_ttc`.

The helper avoids unsafe direct casting and handles missing/malformed extraction data safely.

Representative logic:

```php
public function latestExtraction(): HasOne
{
    return $this->hasOne(Extraction::class)->latestOfMany('version');
}

public function getLatestExtractedTotalTtc(): float
{
    $latestExtraction = $this->resolveLatestExtraction();
    $fields = $latestExtraction?->result_json['fields'] ?? [];
    $value = $fields['total_ttc'] ?? null;

    if (is_string($value)) {
        $value = str_replace(',', '.', trim($value));
    }

    return is_numeric($value) ? (float) $value : 0.0;
}
```

This prevents malformed OCR/extraction values from silently corrupting the financial summary.

## 3.5 Controller response consistency

Both dossier and escalation response paths now reuse the same financial payload helper.

This matters because workflow actions such as:

- escalation;
- return to Claims Manager;
- complement request;
- final processing;

can all change the dossier’s visible summary state. Returning the same financial summary shape from all response paths keeps the frontend stable after mutations.

## 3.6 Frontend summary card update

The dossier summary card was updated to show:

- `Requested Total`;
- `Accepted Total`;
- `Rejected Total`;
- `Final Reimbursable Total`.

The old visible labels were removed from the main card:

- `Current Total`;
- `Display Total`;
- `Stored Total`.

For non-processed dossiers, `Final Reimbursable Total` displays a clean placeholder instead of pretending a final amount exists.

This is more accurate for active review/escalation/complement states.

# 5) Active Workflow Context Consolidation

## 5.1 Problem

`DossierDetail` had multiple places that could explain the current workflow state:

- top banners;
- escalation context block;
- return/complement context areas;
- workflow history.

This created duplication, especially in cases involving:

- supervisor return to Claims Manager;
- supervisor complement request;
- return to preparation;
- escalation review.

The page needed one current-context block, not several competing explanations.

## 5.2 New component: `ActiveWorkflowContextCard`

A new focused component was introduced:

```text
frontend/src/pages/DossierDetail/components/ActiveWorkflowContextCard.jsx
```

Its purpose is narrow:

> Explain why the dossier is currently waiting for action and who needs to act.

It does not replace workflow history. It does not contain action buttons. It does not mutate the dossier.

## 5.3 Rendering rules

The card renders at most once.

The current rules are:

- `RECEIVED`: no active context card;
- normal `IN_PROGRESS`: no active context card;
- `AWAITING_COMPLEMENT + CLAIMS_MANAGER_RETURN`: visible to `AGENT` and `ADMIN`;
- `AWAITING_COMPLEMENT + SUPERVISOR_COMPLEMENT_REQUEST`: visible to `AGENT` and `ADMIN`;
- `UNDER_REVIEW + chef_decision_type = RETURNED`: visible to `CLAIMS_MANAGER` and `ADMIN`;
- `IN_ESCALATION`: visible to `SUPERVISOR` and `ADMIN`, only when real escalation context exists;
- `PROCESSED`: no active context card.

ADMIN bypasses role visibility, but not state relevance. This means ADMIN can see the active context if one exists, but stale cards are not forced into normal or final states.

## 5.4 Removed duplicated UI

The old duplicated top workflow banners were removed from `DossierDetail`:

- pending supervisor review banner;
- returned-by-supervisor banner;
- preparation reopened banner.

`EscalationInfoBlock` was also removed from the active rendering path once its responsibility was absorbed by `ActiveWorkflowContextCard`.

The full immutable history remains handled by `WorkflowHistoryCard`.

# 9) Frontend Shell and Styling Polish

## 9.1 Scrollbar behavior

The previous global scrollbar hiding rule was too aggressive. It hid scrollbars everywhere, including:

- tables;
- modals;
- notification panels;
- long pages;
- workflow history.

The implementation was replaced by a professional scrollbar strategy:

- horizontal overflow remains protected;
- the vertical scrollbar gutter is stabilized;

A small route-aware component was introduced:


It uses the current React Router location and calls `window.scrollTo(0, 0)` on path changes.

This improves SPA navigation without introducing dependencies.

## 9.2 Redundant `App.css` cleanup

`App.css` contained redundant global overrides for:

- buttons;
- cards;
- form controls;
- table hover;

These rules duplicated the centralized design system in:

- `frontend/src/styles/design-tokens.css`;
- `frontend/src/styles/common.css`;
