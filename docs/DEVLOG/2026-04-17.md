## 1) Objective of Day 35

After Day 34, the business workflow and RBAC layer were already stabilized.
The application was functionally strong, but several visible areas still felt slightly inconsistent or generic:

- login and navbar branding still carried generic product wording
- browser-level branding was not fully aligned with the visual identity shown inside the app
- list pagination required repeated next/previous clicks for larger datasets
- toast notifications communicated only technical success/failure, not **business meaning**
- backend success/error messages still leaked old domain terms such as **dossier** and **rubrique** into the UI

Day 35 was therefore treated as a **targeted refinement and commit-readiness sprint**, not as a redesign.
The goal was to make the application feel more enterprise-ready, more coherent for soutenance, and more aligned with the already validated domain language:

- **case file**
- **section**
- **document**
- **claims manager**
- **supervisor**

## 4) Shared Pagination Improvement – “Jump To”

### 4.1 Problem identified

The reusable pagination footer already supported:

- current page indicator
- total count
- previous
- next

However, the supervisor requested a direct “jump to page” capability.
For long internal tables, repeatedly clicking **Next** is weak UX.

### 4.2 Design decision

Instead of introducing heavy numbered pagination across every table, the shared pagination component was extended with a compact direct-navigation control.

This kept the enterprise table footer compact while still improving usability.

### 4.3 Implemented behavior

The new shared footer supports:

- numeric input for target page
- **Go** button
- Enter-key submission
- normalization of out-of-range values
- reuse across all screens already using the shared footer

Representative logic:

```jsx
const normalizeTargetPage = (value) => {
  const parsed = Number.parseInt(String(value).trim(), 10);

  if (Number.isNaN(parsed)) return null;
  if (parsed < 1) return 1;
  if (parsed > lastPage) return lastPage;
  return parsed;
};
```

Representative UI block:

```jsx
<div className="table-pagination-footer__jump-controls">
  <input
    type="number"
    min="1"
    max={lastPage}
    value={jumpPage}
    onChange={(event) => setJumpPage(event.target.value)}
    onKeyDown={handleJumpKeyDown}
  />
  <button type="button" onClick={handleJumpSubmit}>Go</button>
</div>
```

### 4.4 Architectural benefit

Because the feature was implemented in the **shared** pagination component, one change improved all list screens consistently instead of creating page-by-page divergence.

This is a good example of targeted polish done the right way:

- one abstraction
- one behavioral contract
- multiple screens improved

## 5) Semantic Toast Notification Layer

### 5.1 Problem identified

The application already used `react-hot-toast` globally, but page-level mutations still called `toast.success(...)` and `toast.error(...)` directly.

This created an important UX problem:

- **Accept document** and **Reject document** both looked like the same green success event
- **Activate user** and **Deactivate user** both used the same positive feedback style
- workflow-routing actions such as **return to preparation** had no visual distinction from approval actions

Technically the requests succeeded, but the **business meaning** was different.

### 5.2 New semantic layer

A shared helper file was introduced to centralize notification semantics:

- `notifySuccess(...)`
- `notifyDangerSuccess(...)`
- `notifyWorkflowSuccess(...)`
- `notifyError(...)`

This kept `react-hot-toast` as the underlying engine while replacing direct page-level usage with domain-aware helper calls.

Representative implementation:

```js
export const notifyDangerSuccess = (message, options = {}) =>
  toast.success(message, {
    ...BASE_OPTIONS,
    ...dangerVariant(),
    ...options,
  });
```

### 5.3 Tokenized toast styling

Toast colors were not hardcoded randomly inside page files.
Instead, dedicated toast tokens were added to the centralized design-token layer.

Representative token set:

```css
--toast-success-border: #bbf7d0;
--toast-danger-border: #fecaca;
--toast-workflow-border: #bfdbfe;
--toast-error-border: #fecaca;
```

This preserved the frontend architecture direction established earlier:

- shared design tokens remain the source of truth
- shared helper layer controls semantic feedback
- page components focus on orchestration and business actions

### 5.4 Semantic mapping adopted

The final feedback model became:

#### Green success
Used for positive outcomes such as:

- create user
- create section
- attach documents
- validate document
- accept document
- submit case file
- process case file
- activate user
- approve escalation

#### Red destructive success
Used for destructive-but-successful outcomes such as:

- reject document
- reject section
- detach document
- delete section
- delete document
- delete case file
- deactivate user

#### Blue workflow success
Used for state-routing outcomes such as:

- return to claims manager
- request complement
- return to preparation

#### Error
Reserved only for actual failures:

- validation failure
- network failure
- permission failure
- missing required note
- failed mutation

This distinction significantly improved UI honesty.
The application now communicates **what the action means**, not only whether the HTTP request returned 200.

## 6) Toast Migration Across Frontend Workflows

The semantic toast layer was then propagated through the main interaction-heavy screens.

### 6.1 DossierDetail workflow orchestration

`DossierDetail.jsx` became the main orchestration page for semantic workflow feedback.

Examples:

- accepting a document stays green
- rejecting a document becomes danger-styled
- returning the case file to preparation becomes workflow-styled
- escalation approval remains positive and distinct from destructive decisions

Representative call pattern:

```jsx
notifyDangerSuccess(getToastMessage(response, 'Document rejected successfully.'));
notifyWorkflowSuccess(getToastMessage(response, 'Case file returned to preparation.'));
```

### 6.2 Admin user management

`AdminUsers.jsx` was updated to distinguish:

- activation → positive success
- deactivation → danger success

Representative logic:

```jsx
if (statusModal.nextIsActive) {

} else {
  notifyDangerSuccess(successMessage);
}
```

This directly answered the encadrant’s feedback about not using the same green tick for deactivation.

### 6.3 List pages and nested interaction components

The same migration was applied across:

- `DossiersList.jsx`
- `DocumentsList.jsx`
- `DocumentValidation.jsx`
- `RubriquesSection.jsx`

This closed the inconsistency where some pages used the new toast system while others still emitted old generic notifications.

## 7) Backend Wording Normalization for UI Feedback

### 7.1 Problem identified

Even with a correct frontend toast layer, the UI still showed old wording when the backend returned messages such as:

- `Dossier submitted successfully.`
- `Rubrique deleted successfully.`
- `This dossier is frozen and cannot be modified.`

That meant the final visible feedback was still linguistically inconsistent.

### 7.2 Controllers normalized

User-facing messages were normalized in the main workflow controllers:

- `DossierController.php`
- `RubriqueController.php`
- `DossierEscalationController.php`
- `DocumentDecisionController.php`
- `DocumentController.php`

The rule was simple:

- **dossier** → **case file**
- **rubrique** → **section**

Representative examples:

```php
return response()->json([
    'message' => 'Case file submitted successfully.',
], 200);
```

```php
return response()->json([
    'message' => 'Section deleted successfully.',
], 200);
```

### 7.3 Important architectural decision

Only **user-facing response messages** were normalized.
Internal implementation names were intentionally preserved:

- controllers still keep their current class names
- routes and models remain unchanged
- no database columns were renamed
- no API contract keys were broken just for wording purity

This was the correct trade-off:

- visible UX becomes clean
- internal architecture remains stable
- no unnecessary refactor risk is introduced late in the project
