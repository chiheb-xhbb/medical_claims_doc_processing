## 3) Backend â€“ Active User Enforcement

### 3.1 New middleware: `EnsureUserIsActive`

A dedicated middleware was created to enforce account activity on authenticated requests:

- `backend/app/Http/Middleware/EnsureUserIsActive.php`

### 3.2 Middleware behavior

For every authenticated request, the middleware now:

1. retrieves the authenticated user
2. checks `is_active`
3. if inactive:
   - revokes the current Sanctum token
   - returns a JSON error response
   - blocks access immediately

### 3.3 Why this was necessary

Blocking login alone is not sufficient in a real RBAC system.

Without middleware enforcement, a user deactivated by an administrator could still continue using an already-issued token. Day 22 therefore added true active-session eviction behavior so that deactivation becomes effective immediately, not only on the next login attempt.

### 3.4 Laravel registration of the middleware

The middleware was registered in `Kernel.php` under the alias:

```php
'active.user' => EnsureUserIsActive::class
```

Authenticated route groups were then updated to use:

```php
['auth:sanctum', 'active.user']
```

This ensured that all protected API access now respects the account activity state.

### 4.3 Closed-platform direction

## 8) Backend â€“ Role Update and Status Update Safety

### 8.1 Role update endpoint

`PATCH /api/admin/users/{user}/role` allows an administrator to update the role of another user.

### 8.2 Status update endpoint

`PATCH /api/admin/users/{user}/status` allows an administrator to activate or deactivate a user account.

### 8.3 Self-protection rules enforced

The backend now prevents unsafe administrative actions such as:

- self-deactivation
- self-removal of the admin role

### 8.4 Last active admin protection

A critical protection was added to avoid locking the platform out of administrative access:

- the last active admin cannot be downgraded
- the last active admin cannot be deactivated

These checks were implemented inside database transactions with row locking to avoid race conditions during concurrent updates.

### 8.5 Token revocation on deactivation

When an account is deactivated, all Sanctum tokens for that user are revoked. Combined with the active-user middleware, this guarantees immediate session invalidation.
