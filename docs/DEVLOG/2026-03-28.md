## 1) Objective of Day 21

After Day 20, the dossier workflow was already usable from the browser and aligned with the business structure:

```text
Dossier
  └── Rubriques
        └── Documents
```

However, the application still lacked a fully coherent role model in practice. The workflow was mostly status-driven, while the real business need required explicit separation between:

- **Agent** → preparation and submission
- **Gestionnaire** → technical control and business review
- **Admin** → global supervision

Day 21 therefore focused on transforming the application from a simple authenticated system into a **role-aware system**, with RBAC applied both in the backend and in the frontend.

The main targets of the day were:
- add a `role` field to users
- expose the role in the authentication payload
- secure dossier actions by role in Laravel
- secure document access and validation by role in Laravel
- adapt the React UI and routes to the same role model
- validate the complete role behavior in browser and Postman

## 2) Backend Role Foundation

### 2.1 New `UserRole` enum
A dedicated `UserRole` enum was introduced to formalize the three supported business roles:

- `AGENT`
- `GESTIONNAIRE`
- `ADMIN`

This avoids fragile string checks spread everywhere in the code and gives a stable foundation for future authorization rules.

### 2.2 Database migration for user roles
A new migration added the `role` column to the `users` table with a default value of `AGENT`.

This decision ensures:
- backward compatibility for existing users
- a safe default role for newly registered accounts
- a clear entry point for future admin-side user management

### 2.3 `User` model update
The `User` model was updated to:
- include the `role` field in the fillable/cast logic
- cast `role` through `UserRole`
- preserve existing authentication and document relationships

This makes the user role available as a first-class domain attribute across the application.

---

## 3) Authentication Refactor

### 3.1 `AuthController` update
The authentication controller was refactored so that `register`, `login`, and `me` now return a stable user payload containing:

- `id`
- `name`
- `email`
- `role`

This was a critical step because frontend role-aware behavior depends on receiving a reliable role value immediately after authentication.

### 3.2 Default registration behavior
New users created through the registration flow are assigned the default role `AGENT`.

This keeps the registration flow simple and safe while reserving elevated roles to controlled assignment.

## 4) Backend RBAC – Dossier Workflow

### 4.1 `DossierController` role-aware protection
`DossierController` was refactored to enforce role-based access on dossier operations.

Implemented logic:

- **AGENT / ADMIN**
  - create dossier
  - update own dossier (Agent) or any dossier (Admin)
  - delete dossier where allowed by status rules
  - submit dossier for review

- **GESTIONNAIRE / ADMIN**
  - process dossier
  - access review-oriented dossier flow

### 4.2 Ownership rule for Agent
The agent remains limited to dossiers they created themselves.

This preserves the preparation ownership logic while avoiding cross-agent interference.

### 4.3 Review visibility for Gestionnaire
The gestionnaire can view dossiers relevant to the review lifecycle, especially those under validation or already finalized.

This aligns the backend with the actual review responsibility and avoids exposing irrelevant preparation drafts.

---

## 5) Backend RBAC – Rubriques and Decisions

### 5.1 `RubriqueController`
The rubrique controller was aligned with the new role model.

Implemented behavior:

- **AGENT / ADMIN**
  - create rubrique
  - update rubrique
  - delete rubrique
  - attach validated documents
  - detach documents

- **GESTIONNAIRE / ADMIN**
  - reject an entire rubrique during review

### 5.2 Dossier transition to `IN_PROGRESS`
The controller preserves the intended business transition:

```text
RECEIVED -> IN_PROGRESS -> TO_VALIDATE -> PROCESSED
```

The first rubrique creation moves the dossier from `RECEIVED` to `IN_PROGRESS`, keeping the state model coherent.

### 5.3 `DocumentDecisionController`
The document decision controller was refactored so that acceptance and rejection are no longer owner-based operations.

Implemented behavior:

- **GESTIONNAIRE / ADMIN** can:
  - accept a document at business level
  - reject a document at business level

- decisions are allowed only when:
  - the dossier is in `TO_VALIDATE`
  - the dossier is not frozen
  - the technical document status is compatible with review

The controller preserves transactional integrity and keeps rubrique status refresh and dossier total refresh inside the same business flow.

---

## 6) Backend RBAC – Document Module

### 6.1 `DocumentController` role-aware listing
The original document listing was still owner-scoped. This was correct for agents, but inconsistent with the chosen business rule for gestionnaires.

The document controller was therefore updated with the following behavior:

- **AGENT** → sees only their own documents
- **GESTIONNAIRE** → sees documents globally
- **ADMIN** → sees documents globally

### 6.2 Document detail access
The same role-aware logic was applied to `show()` so that the validation page can be opened consistently by roles allowed to work on document validation.

### 6.3 Upload protection via `StoreDocumentRequest`
Upload authorization is now enforced at backend level directly inside `StoreDocumentRequest`.

Implemented rule:
- **AGENT / ADMIN** → upload allowed
- **GESTIONNAIRE** → upload forbidden

This guarantees that even if a frontend route is bypassed manually, unauthorized upload attempts are still blocked by Laravel.

### 6.4 `DocumentValidationController`
The technical validation controller was updated so that validation is no longer limited to the document owner.

Final rule:
- **AGENT** → can validate own documents
- **GESTIONNAIRE** → can validate documents
- **ADMIN** → can validate documents

This matches the final business decision taken during Day 21: the gestionnaire participates in technical document control, but not in initial upload.

## 7) Frontend RBAC – Authentication and Session State

### 7.1 `auth.js` role persistence
The frontend authentication service was upgraded to normalize and persist the authenticated user role in local storage.

It now provides:
- `getStoredUser()`
- `getStoredRole()`
- role normalization on login and `/me`
- `AUTH_CHANGED_EVENT` dispatching for UI synchronization

### 7.2 Role-aware landing logic
A small role-aware landing helper was introduced so that the default entry page becomes coherent with the role:

- **AGENT** → `/documents`
- **GESTIONNAIRE** → `/dossiers`
- **ADMIN** → `/dossiers`

This removed the earlier one-size-fits-all redirection behavior.

### 7.3 `api.js` improved error handling
The Axios service was improved to surface clearer backend messages depending on status code:

- `401` → session/auth problem
- `403` → permission issue
- `404` → resource not found
- `422` → business rule violation

This makes role-related denials more understandable during manual testing and normal usage.

---

## 8) Frontend RBAC – Route Protection and Navigation

### 8.1 `ProtectedRoute.jsx`
The route wrapper was extended to support `allowedRoles`.

This allows frontend route-level protection in addition to backend enforcement.

Examples implemented:
- `/documents/upload` → only `AGENT` and `ADMIN`
- `/dossiers/create` → only `AGENT` and `ADMIN`

### 8.2 Forbidden redirect feedback
When a role attempts to access a forbidden page manually, the app now redirects safely and shows a short warning message instead of failing silently.

### 8.3 `Navbar.jsx`
The navigation bar was refactored to become role-aware.

Implemented behavior:
- **AGENT** sees `My Dossiers`
- **GESTIONNAIRE** sees `Dossiers to Review`
- **ADMIN** sees `All Dossiers`

Upload visibility was also aligned with the role rules:
- hidden for `GESTIONNAIRE`
- visible for `AGENT` and `ADMIN`

---

## 9) Frontend RBAC – Dossier and Document Pages

### 9.1 `DossiersList.jsx`
The dossier list page now adapts its:
- page title
- subtitle
- empty state
- create button visibility

according to the authenticated role.

This gives each role a more coherent entry point without creating multiple separate pages.

### 9.2 `DossierDetail.jsx`
The dossier detail page was upgraded from a mainly status-driven screen to a **role + status-driven** screen.

Implemented behavior:

**Preparation actions**
- visible only for `AGENT / ADMIN`
- create rubrique
- attach documents
- detach documents
- submit dossier

**Review actions**
- visible only for `GESTIONNAIRE / ADMIN`
- accept document
- reject document
- reject rubrique
- process dossier

**Frozen states**
- `PROCESSED` and `EXPORTED` remain read-only

This is the main frontend expression of the Maker–Checker logic introduced during Day 21.

### 9.3 `DocumentsList.jsx`
The documents page was aligned with the final document policy:

- **AGENT** → sees own documents and upload UI
- **GESTIONNAIRE** → sees documents but no upload button or upload CTA
- **ADMIN** → sees all documents and upload UI

Retry behavior remains owner-aware for failed technical documents, which keeps the technical recovery flow conservative.

### 9.4 `Login.jsx`
The login page now redirects users to the role-appropriate landing page after successful authentication.

## 11) Architecture Validation

**Day 21 role-aware application flow:**

```text
React Frontend
   ↓
Role-aware navigation, routes, and UI gating
   ↓
Laravel API
   ↓
Role-aware controllers and requests
   ↓
MySQL
```

### Compliance with project rules
- React still communicates only with Laravel
- Laravel remains the source of truth for role enforcement and workflow rules
- FastAPI remains unchanged and stateless
- technical document validation remains separated from dossier business review
- dossier business workflow stays centered on `Dossier -> Rubriques -> Documents`

## 13) System State at End of Day 21

At this point, the application no longer relies only on authentication and dossier status for access behavior. It now includes a first complete RBAC layer with coherent behavior across:

- authentication payloads
- protected routes
- navigation
- document access
- technical validation
- dossier preparation
- dossier review
- admin supervision

The system now expresses the business roles in a much more explicit and defensible way, both in code and in browser behavior.

## 14) Known Limitation

A full **admin user management module** is not implemented yet.

This means the application already supports the role model technically, but the admin still does not yet have a dedicated UI to:
- create users
- update user roles
- activate/deactivate users
- delete users

Role assignment is therefore still handled manually at development level for now.
