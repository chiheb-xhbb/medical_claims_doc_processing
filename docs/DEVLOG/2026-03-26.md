## 1) Objective of Day 19

After Day 18, the platform already supported a first dossier workflow from the browser:

- create dossier
- attach validated documents
- compute dossier total
- display dossier detail

However, this design was still based on a simplified structure where a dossier grouped documents directly and the final workflow remained too coarse at dossier level.

Following the supervisor’s clarification, the expected business model became more granular:

- a **dossier** contains one or more **rubriques**
- a **rubrique** contains one or more **documents**
- a whole rubrique can be rejected
- a single document inside a rubrique can be rejected without rejecting the whole rubrique
- the dossier final state should be generic rather than “validated / rejected”
- the final reimbursable total should be based only on accepted documents

Day 19 focused on refactoring the backend architecture to align the implementation with this clarified business logic.

## 2) Business Model Refactor

### 2.1 New target structure

The dossier workflow was refactored from:

```text
Dossier
  └── Documents
```

to:

```text
Dossier
  └── Rubriques
        └── Documents
```

This change makes it possible to represent two distinct business scenarios:

* rejection of an entire rubrique with all included documents
* rejection of a single document inside a rubrique while keeping the rubrique itself partially valid

### 2.2 Separation of technical and business statuses

A key architectural decision was preserved:

* **DocumentStatus** remains the technical lifecycle:

  * `UPLOADED`
  * `PROCESSING`
  * `PROCESSED`
  * `VALIDATED`
  * `FAILED`

* **DocumentDecisionStatus** now represents the reimbursement decision:

  * `PENDING`
  * `ACCEPTED`
  * `REJECTED`

This keeps OCR/validation pipeline state separate from reimbursement review state.

## 3) Enumeration Refactoring

### 3.1 DossierStatus normalized to English

The dossier enum was refactored to use consistent English values across the backend:

* `RECEIVED`
* `IN_PROGRESS`
* `TO_VALIDATE`
* `PROCESSED`
* `EXPORTED`

This replaces the older mixed French business states and aligns dossier internals with the rest of the codebase.

### 3.2 New RubriqueStatus

Created `RubriqueStatus` with:

* `PENDING`
* `ACCEPTED`
* `REJECTED`
* `PARTIAL`

Purpose:

* `PENDING` → no final review decision yet
* `ACCEPTED` → all relevant documents accepted
* `REJECTED` → whole rubrique rejected
* `PARTIAL` → mixed accepted / rejected documents

### 3.3 New DocumentDecisionStatus

Created `DocumentDecisionStatus` with:

* `PENDING`
* `ACCEPTED`
* `REJECTED`

This supports document-by-document review inside each rubrique.

## 4) Database Schema Evolution

### 4.1 New `rubriques` table

Created a dedicated `rubriques` table with the following fields:

* `id`
* `dossier_id`
* `title`
* `status`
* `notes`
* `created_by`
* timestamps

This establishes the missing intermediate business entity between dossier and document.

### 4.2 Documents table enrichment

Extended `documents` table with:

* `rubrique_id`
* `decision_status`
* `decision_by`
* `decision_at`
* `decision_note`

These fields enable:

* linking each document to a rubrique
* storing reimbursement review decisions
* tracking who made the decision and when

### 4.3 Backward compatibility decision

The existing `documents.dossier_id` field was intentionally kept temporarily during the refactor to avoid breaking the whole workflow at once.

This allowed a progressive transition toward the new hierarchy while keeping previously built dossier logic adaptable.

## 6) Backend Controller Layer

### 6.1 `RubriqueController`

Created a new API controller to handle rubrique-level actions:

* `store()`
* `update()`
* `destroy()`
* `attachDocuments()`
* `detachDocument()`
* `rejectAll()`

Key rules enforced:

* ownership checks
* dossier frozen-state checks
* validated document requirement before attachment
* prevention of deleting rubriques that already contain documents
* total refresh after attach/detach/reject actions

### 6.2 `DocumentDecisionController`

Created a dedicated controller for document-by-document decisions:

* `accept()`
* `reject()`

This layer handles:

* validation that the document belongs to a rubrique
* dossier frozen-state protection
* requirement that only `VALIDATED` documents can be accepted/rejected
* persistence of:

  * `decision_status`
  * `decision_by`
  * `decision_at`
  * `decision_note`

It also refreshes rubrique status and dossier totals after each decision.

### 6.3 `DossierController` refactor

Refactored `DossierController` to support the new business flow:

* `index()` updated to return `rubriques_count` and `documents_count`
* `show()` now loads:

  * `creator`
  * `rubriques.creator`
  * `rubriques.documents.extractions`
  * `rubriques.documents.decisionMaker`
* `store()` creates dossier with `RECEIVED`
* `update()` blocks modification when dossier is frozen
* `destroy()` respects deletion rules
* `submit()` introduced to move dossier to `TO_VALIDATE`
* `process()` introduced to finalize dossier as `PROCESSED`

The old direct `attachDocuments()` flow on the dossier itself was effectively replaced by rubrique-level attachment.

## 8) Data Consistency Fixes During Refactor

### 8.1 Old dossier status values in database

A compatibility problem was identified after refactoring the enum to English values:

existing rows still contained older values such as:

* `RECU`
* `EN_TRAITEMENT`
* `A_VALIDER`
* `VALIDE`
* `REJETE`
* `EXPORTE`

These values caused enum-casting failures after the PHP enum was updated.

The database values were corrected to match the new enum states.

### 8.2 Ambiguous `status` column in total calculation

After switching to `hasManyThrough(Document::class, Rubrique::class)`, queries on `$this->documents()` produced SQL ambiguity because both `documents` and `rubriques` contain a `status` column.

This was fixed by explicitly qualifying:

* `documents.status`
* `documents.decision_status`

inside dossier total and processing logic.

## 10) Final Workflow Achieved

At the end of Day 19, the backend now supports this dossier review flow:

```text
Create dossier
   ↓
Create rubrique
   ↓
Attach validated document(s)
   ↓
Dossier moves to IN_PROGRESS
   ↓
Submit dossier
   ↓
Dossier moves to TO_VALIDATE
   ↓
Accept / reject documents
   ↓
Rubrique status recalculated
   ↓
Process dossier
   ↓
Dossier moves to PROCESSED
   ↓
Final total frozen
```

This is the first time the dossier module fully reflects the supervisor’s clarified business model.

## 11) Architecture State After Day 19

The system now distinguishes three levels clearly:

### Technical document workflow

Handled by the existing OCR/HITL pipeline:

```text
UPLOADED → PROCESSING → PROCESSED → VALIDATED
```

### Business review workflow

Handled by the new decision layer:

```text
PENDING → ACCEPTED / REJECTED
```

### Dossier lifecycle

Handled by the business aggregate:

```text
RECEIVED → IN_PROGRESS → TO_VALIDATE → PROCESSED → EXPORTED
```

This separation significantly improves the conceptual clarity of the system.
