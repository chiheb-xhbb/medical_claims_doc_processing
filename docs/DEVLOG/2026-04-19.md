# 1) Objective of Day 37

After Day 36, the application already had:

- a stabilized dossier workflow;
- a persistent Laravel-backed notification subsystem;
- a professional navbar shell;
- semantic toasts and normalized workflow wording;
- a coherent role model across `AGENT`, `CLAIMS_MANAGER`, `SUPERVISOR`, and `ADMIN`.

However, one important presentation-layer gap still remained:

- the UI was still effectively monolingual;
- business wording was scattered across raw strings and helper constants;
- some screens were partially localized while others still leaked English-only labels;
- a future language switch would have been fragile because internal values and visible labels were still too tightly coupled in several places.

Day 37 therefore introduced a **real frontend internationalization pass**, not as a cosmetic patch, but as a controlled presentation-layer enhancement designed to keep the technical architecture stable while making the UI bilingual and more academically defendable.

The main goal was to support:

- **English** as the default UI language;
- **French** as a real second language;
- centralized translation dictionaries;
- controlled business terminology;
- translation-safe badges, labels, notifications, and workflow surfaces;
- a polished language switcher integrated into both the navbar and the login page;
- zero regressions in authentication, routing, workflow screens, and notification behavior.

# 2) Architecture and Scope Decisions

The day was guided by a few strict architectural decisions to avoid turning i18n into a destabilizing late-stage refactor.

## 2.1 Internal values remain unchanged

The project deliberately keeps backend and internal frontend constants stable.

That means values such as:

- `AGENT`
- `CLAIMS_MANAGER`
- `SUPERVISOR`
- `ADMIN`
- `IN_PROGRESS`
- `UNDER_REVIEW`
- `IN_ESCALATION`
- `AWAITING_COMPLEMENT`
- `PENDING`
- `ACCEPTED`
- `REJECTED`

remain unchanged in:

- Laravel controllers and responses;
- route protection;
- workflow logic;
- service calls;
- persisted domain data;
- internal React constants.

The UI layer alone becomes multilingual.

This preserves the clean separation already established earlier in the project:

- Laravel owns business truth;
- React renders presentation;
- i18n affects labels, not domain state.

## 2.2 No fake third language

Arabic was explicitly **not** added during this pass.

The decision was intentional:

- the repo was already coherently wired for EN/FR;
- adding an `AR` button without full locale resources and directionality support would have created a fake feature;
- a partial Arabic option would make the project look less professional, not more.

The final implementation therefore exposes only the languages that are genuinely supported end-to-end:

- **EN**
- **FR**

## 2.3 No backend, database, or FastAPI refactor

This implementation remains fully frontend-scoped.

No changes were introduced in:

- Laravel controllers,
- migrations,
- model structures,
- FastAPI OCR behavior,
- API payload contracts.

That was the right decision because the problem to solve was **presentation consistency**, not business-logic redesign.

# 3) Controlled Business Vocabulary

One of the most important goals of Day 37 was not simple string translation, but **controlled domain wording**.

The project already had a canonical internal vocabulary. Day 37 made the visible UI respect it consistently across languages.

## 3.1 Role wording

The following visible mappings were enforced:

- `AGENT` → **Agent** / **Agent**
- `CLAIMS_MANAGER` → **Claims Manager** / **Gestionnaire**
- `SUPERVISOR` → **Supervisor** / **Chef hiérarchique**
- `ADMIN` → **Administrator** / **Administrateur**

This is especially important because French literal translation such as *Superviseur* would be less faithful to the business meaning than *Chef hiérarchique*.

## 3.2 Domain nouns

Canonical UI wording was also stabilized for the main entities:

- **Case File** / **Dossier**
- **Case Files** / **Dossiers**
- **Section** / **Rubrique**
- **Sections** / **Rubriques**
- **Document** / **Document**
- **Documents** / **Documents**

## 3.3 Workflow statuses

The user now sees translated human labels rather than raw technical values.

For example:

- `IN_PROGRESS` → **In Progress** / **En cours de préparation**
- `UNDER_REVIEW` → **Under Review** / **En cours de révision**
- `IN_ESCALATION` → **In Escalation** / **En escalade**
- `AWAITING_COMPLEMENT` → **Awaiting Complement** / **En attente de complément**

This makes the system much easier to defend in front of a jury, because the visible labels now reflect the real business meaning while the code remains stable underneath.

## 4.2 Translation dictionaries

Two complete locale dictionaries were introduced:

- `frontend/src/i18n/locales/fr/common.json`

These files centralize translation keys for:

- navigation;
- roles;
- dossier and document statuses;
- notifications;
- list pages;
- detail pages;
- modals;
- upload flow;
- admin actions;
- shared feedback components.

## 4.3 Language persistence

The selected language is persisted in local storage through the existing `app_language` key, so the user’s language choice survives refresh and reconnect.

Representative behavior:

```jsx
const handleChangeLanguage = (language) => {
  i18n.changeLanguage(language);
  localStorage.setItem('app_language', language);
};
```

This keeps the feature practical rather than decorative.

# 5) Translation-Safe Label Layer

A key weakness in the earlier UI was the presence of static English label maps inside helper files.

## 5.1 `domainLabels.js` refactor

`frontend/src/constants/domainLabels.js` was refactored so that it still preserves internal constants, but resolves visible labels at render time through i18n-aware helpers.

Representative pattern:

```js
export function getRoleLabel(role) {
  return i18n.t(`roles.${role}`, { defaultValue: formatEnumFallback(role) });
}

export function getDossierStatusLabel(status) {
  return i18n.t(`dossierStatus.${status}`, { defaultValue: formatEnumFallback(status) });
}
```

## 5.2 Why this matters

Without this change, language switching would remain fragile because components would keep reading static English labels even after the locale changed.

The final label layer now guarantees:

- stable internal values;
- translated visible text;
- safe fallback behavior;
- no need to rewrite the whole domain model.

# 6) Raw Enum Leak Elimination

One of the most important quality checks of Day 37 was ensuring that raw backend-like values do **not** appear directly in visible UI.

## 6.1 Problem identified

Several status and decision surfaces still risked falling back to text derived from enum values such as:

- `IN_PROGRESS`
- `UNDER_REVIEW`
- `IN_ESCALATION`

Even when transformed with `replace(/_/g, ' ')`, this still produces a technical label rather than a true business label.

## 6.2 Final behavior

Badges and helper components were hardened so they now rely on:

- translated label resolvers first;
- human-readable fallback labels only if absolutely necessary;
- no direct raw-enum leak path in visible UI.

This hardening affected especially:

- `StatusBadge.jsx`
- `CaseFileStatusBadge.jsx`
- `DecisionBadge.jsx`
- `UserRoleBadge.jsx`
- `AccountStatusBadge.jsx`
- and any component depending on `domainLabels.js`

Representative badge behavior:

```jsx
const label = getDossierStatusLabel(normalizedStatus);
```

This is a small but important quality marker: the app now looks designed for users, not for developers.
