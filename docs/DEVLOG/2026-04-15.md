# 1) Objective of Day 33

After Day 32, the platform already supported the core operational flow:

- dossier preparation and review
- escalation and supervisor decisions
- secure source-document access
- OCR/extraction + human validation
- RBAC and protected routes
- self password change and admin password reset

However, one major account-recovery gap remained:

> A logged-out user still had no self-service password recovery path by email.

That meant the application could handle:

- login/logout
- self password change (when already authenticated)
- admin password reset

but not the real-world case of:

- user forgets password
- user is not logged in
- user needs to recover access without admin intervention

Day 33 therefore focused on a **production-like password recovery flow**, not on a generic auth redesign.

The implementation targeted the following outcomes:

- add a public **forgot-password** endpoint
- add a public **reset-password** endpoint
- use Laravel’s native password broker and token flow
- send the reset email to the **React SPA reset page**, not a Laravel Blade page
- preserve neutral responses so account existence is not leaked
- keep invalid token and invalid user errors collapsed into the same user-facing message
- add focused frontend pages for forgot/reset password
- validate the full end-to-end browser flow

# 2) Core Decisions Frozen on Day 33

Before implementation, the recovery flow was intentionally constrained.

## 2.1 Use Laravel reset links and tokens, not OTP/code

Day 33 explicitly **did not** implement:

- email OTP
- numeric verification code
- ad hoc temporary password logic
- custom reset-token persistence

Instead, the implementation used Laravel’s built-in password broker.

That was the right choice because it provides:

- a mature token lifecycle
- existing password reset persistence
- cleaner Laravel-native behavior
- lower security risk than inventing a custom recovery protocol

---

## 2.2 The email must open the React SPA, not Laravel web UI

A critical architecture rule was preserved:

```text
React → Laravel only
```

Since the frontend is a React SPA and the backend is an API, the reset email could not point to a Laravel Blade reset form.

So the reset email was forced to target the frontend route shape:

```text
http://localhost:5173/reset-password?token=...&email=...
```

This became the main technical success criterion of the day.

---

## 2.3 Forgot-password response must not leak account existence

The public recovery entry point was designed to remain neutral.

So the forgot-password endpoint always returns the same success response:

```json
{
  "message": "If this email is registered, a password reset link has been sent."
}
```

This prevents a caller from using the endpoint to enumerate valid accounts.

---

## 2.4 Invalid token and invalid user must produce the same frontend error

During password reset, Laravel can distinguish multiple internal failure reasons, including:

- invalid token
- invalid user

Exposing that distinction publicly would weaken the recovery flow.

So Day 33 enforced one shared user-facing message:

```text
This password reset link is invalid or expired.
```

That rule was implemented in the backend controller layer.

---

## 2.5 Email query param must be URL-safe

The frontend reset link includes the email in the query string.

That means addresses containing characters like `+` must not break parsing.

So the custom notification uses URL encoding when building the reset URL.

This is a small but important correctness detail for real-world email addresses.

## 3.2 Mail configuration prepared for local development

A local SMTP configuration was set in `.env` so Laravel notifications could send recovery emails during development.

The key requirements for the day were:

- working SMTP transport
- valid sender address/name
- frontend URL value exposed to the backend

The backend also needed a dedicated frontend config entry:

```php
'frontend_url' => env('FRONTEND_URL', 'http://localhost:5173'),
```

This allowed runtime code to read:

```php
config('app.frontend_url')
```

instead of pulling `env()` directly inside controller/notification logic.

### Why this mattered

Without this config key, the custom notification could not reliably generate the SPA reset URL.

After `.env` changes, Laravel config and cache were cleared and the local server was restarted.

This was necessary because stale config values would otherwise keep the old mail settings or missing frontend URL in memory.

# 4) Backend – Public Password Recovery Endpoints

## 4.1 `AuthController` extended with `forgotPassword()`

A new public method was added to:

- `backend/app/Http/Controllers/Api/AuthController.php`

### Behavior implemented

The method:

- validates `email`
- calls Laravel’s password broker:

```php
Password::sendResetLink([
    'email' => $request->input('email'),
]);
```

- always returns the same neutral response

### Final response behavior

Even if the user email does not exist, the API still returns:

```json
{
  "message": "If this email is registered, a password reset link has been sent."
}
```

### Why this belongs in `AuthController`

The controller already owned:

- login
- logout
- current user retrieval
- self password change

So account-recovery entry points naturally belong to the same auth responsibility area.

---

## 4.2 `AuthController` extended with `resetPassword()`

A second public method was added to the same controller.

### Validation implemented

The method validates:

- `token`
- `email`
- `password`
- `password_confirmation` through Laravel’s `confirmed` rule

### Reset behavior

The controller calls Laravel’s password broker reset flow:

```php
$status = Password::reset(
    [
        'email' => $validated['email'],
        'password' => $validated['password'],
        'password_confirmation' => $request->input('password_confirmation'),
        'token' => $validated['token'],
    ],
    function (User $user, string $password) {
        $user->forceFill([
            'password' => Hash::make($password),
            'remember_token' => Str::random(60),
        ])->save();

        $user->tokens()->delete();
    }
);
```

### Important result

On successful reset:

- the new password is hashed and persisted
- the remember token is rotated
- existing Sanctum tokens are revoked

Revoking tokens is a useful hardening step in this token-based application because a reset should invalidate prior authenticated sessions.

---

## 4.3 Unified invalid-link error handling

The controller explicitly collapses:

- `Password::INVALID_TOKEN`
- `Password::INVALID_USER`

into the same response:

```json
{
  "message": "This password reset link is invalid or expired."
}
```

This preserves the security rule frozen at the start of the day.

# 5) Backend – Public Route Registration

The recovery routes were added in:

- `backend/routes/api.php`

as public endpoints:

```php
Route::post('/forgot-password', [AuthController::class, 'forgotPassword'])->middleware('throttle:5,1');
Route::post('/reset-password', [AuthController::class, 'resetPassword'])->middleware('throttle:5,1');
```

### Important architectural correction

These routes were intentionally kept **outside**:

```php
Route::middleware(['auth:sanctum', 'active.user'])->group(...)
```

because the recovery flow must work for a logged-out user.

### Additional hardening

Rate limiting was retained on both endpoints.

# 6) Backend – Custom Reset Notification

This was the most important backend block of Day 33.

## 6.1 Problem with default Laravel behavior

By default, Laravel’s password reset notification often assumes a server-rendered reset page.

That would have violated the project architecture because the actual reset UI belongs to the React SPA.

## 6.2 New notification class introduced

A dedicated notification was created:

- `backend/app/Notifications/ResetPasswordNotification.php`

### Final behavior

The notification builds the frontend URL using:

- `config('app.frontend_url')`
- the broker token
- the URL-encoded account email
- the configured broker expiry for the helper message

Core logic:

```php
$frontendUrl = rtrim((string) config('app.frontend_url'), '/');
$email = urlencode($notifiable->getEmailForPasswordReset());
$expireMinutes = (int) config('auth.passwords.users.expire', 60);

$url = $frontendUrl
    . '/reset-password?token=' . $this->token
    . '&email=' . $email;
```

### Final email content

The notification now sends a clear recovery email with:

- subject: `Carte Assurances - Password Reset`
- page-specific call to action
- expiry line based on actual broker config
- React reset URL in the action button and fallback text

---

## 6.3 `User` model override added

To make Laravel use the custom notification instead of the default one, the `User` model was updated with:

```php
public function sendPasswordResetNotification($token)
{
    $this->notify(new ResetPasswordNotification($token));
}
```

### Important bug encountered and fixed

An initial version used a typed signature:

```php
public function sendPasswordResetNotification(string $token): void
```

That caused a PHP compatibility error because Laravel’s parent method is declared without those type declarations.

The override was corrected to match Laravel’s expected signature exactly.

### Result

The custom notification is now used cleanly and without runtime conflicts.

# 7) Frontend – Auth Service Layer Extension

`frontend/src/services/auth.js` was extended with two new helpers.

## 7.1 `forgotPassword(email)`

```js
export async function forgotPassword(email) {
  const response = await api.post('/forgot-password', { email });
  return response.data;
}
```

## 7.2 `resetPassword(...)`

```js
export async function resetPassword({ token, email, password, passwordConfirmation }) {
  const response = await api.post('/reset-password', {
    token,
    email,
    password,
    password_confirmation: passwordConfirmation,
  });

  return response.data;
}
```

### Why this matters

This preserves the existing frontend architecture:

- pages do not make ad hoc Axios calls directly
- auth-related API behavior remains centralized in the auth service layer
- React still talks only to Laravel
